NuttX 任务创建与退出全流程详解

本文针对 NuttX 深入讲解任务的诞生、运行、消亡与回收全链路——TCB 与 task_group 的建立关系、task_init/task_fork/task_spawn/task_execve 四种创建路径、task_activate 入就绪、task_exit 与 task_terminate 两种退出路径、wait/waitid/waitpid 三种回收、SIGCHLD 投递、exit 时 fd 表/信号/sem/mqueue/timer 的回收链。

全部源码引用对应本仓库(nuttx/)里的真实文件，行号即真实行号。

与调度文档不重叠的边界：调度文档讲的是"任务在跑起来后"的事——TCB 已存在、状态机怎么转、就绪/阻塞队列怎么组织、PendSV/SVC 怎么触发切换、上下文切换汇编细节、SMP 派发。本文讲的是"任务怎么进来又怎么出去"——TCB 与 group 从无到有、四种创建路径的差异、首次激活入就绪、退出回收全链、wait/SIGCHLD 闭环。

1. 概览：任务的四种诞生路径与一种死亡

1.1 NuttX 里"任务"是什么

NuttX 用统一的 struct tcb_s（include/nuttx/sched.h:571）表达所有可调度实体：内核任务、应用任务、pthread、idle task。每个 TCB 持有：

  • pid — 唯一 ID
  • group — 指向共享的 struct task_group_s
  • start / entry — 启动函数与入口
  • task_state — 当前状态（衔调度文档 §3）
  • stack_base_ptr / adj_stack_size — 栈
  • waitdog — 软件看门狗实例（衔时钟与定时器文档，所有 timed wait 用它）
  • 信号字段三队列（衔信号文档）
关键区分：tcb 是"线程"语义，group 是"进程"语义。同 group 的多个 tcb 共享 fd 表、信号处置表、环境变量、内存映射、定时器——这就是 pthread 同进程多线程共享资源的根源。fork 复制 group、新建 tcb；pthread 共享 group、新建 tcb。

2.3 struct child_status_s（include/nuttx/sched.h:359）

struct child_status_s
{
  FAR struct child_status_s *flink;   /* 子状态链 */
  uint8_t ch_flags;                    /* CHILD_FLAG_* */
  pid_t   ch_pid;                      /* 子任务 ID */
  int     ch_status;                   /* 子退出状态 */
};

挂在父 group 的 tg_children 链上。子任务 exit 时建一个 child_status_s 节点存其退出码，父 wait 时摘下取码——这是 POSIX wait 回收的实质。

ch_flags 含 CHILD_FLAG_TTYPE 三类（kernel/pthread/task）、CHILD_FLAG_EXITED（已退出待收）、CHILD_FLAG_RESTART（可 restart）。

3.3 栈分配与 up_initial_state 伪造现场

nxtask_setup_stackargs 是关键——它做三件事：

  1. 分配栈：按 ttype 选 kmm（内核）或 umm（用户）。stack_alloc_ptr 存原始分配指针（释放用），stack_base_ptr 存调整后栈顶（去掉 arch frame），adj_stack_size 存调整后大小。
  2. argv 复制到栈：argv 字符串和 argv[] 指针数组都压在栈顶——任务 entry 看到的 argv 是栈上的副本，与调用方的 argv 解耦。
  3. up_initial_state 伪造现场（arch/*/src/* arch 钩子）：
    • xcp.regs[REG_PC] = entry — 入口 PC
    • xcp.regs[REG_LR] = task_exit 或 _exit — LR 设成 task_exit，防止任务函数 return 漏走
    • xcp.regs[REG_SP] = stack_base_ptr — 伪造栈顶
    • R0 = arg（ARM 第一参）、R4-R11 清零
    • 首次切入时 PendSV 的 ldmia xcp.regs 恢复这些伪造值，bx EXC_RETURN 就退到 entry——看起来跟正常切入一模一样

3.4 nxtask_activate() 入就绪（task_activate.c）

void nxtask_activate(FAR struct tcb_s *tcb)
{
  /* 设就绪状态 */
  tcb->task_state = TSTATE_TASK_READY;

  /* 入就绪队列 */
  nxsched_add_readytorun(tcb);   /* 衔调度文档 §4 */

  /* 若新任务优先级高于当前，触发 PendSV 切换 */
  nxsched_mergepending();   /* 或 sched_unlock 时再 merge */
}

新建任务的首次"切入"靠 PendSV——伪造现场被 ldmia 恢复后 bx 退到 entry。衔调度文档 §7.3：第一次切入与正常切入看起来一模一样，区别只在 xcp.regs 是 up_initial_state 伪造的而非真现场。

4. 路径 ②：fork —— 复制 group 的全链路

4.1 fork 与 task_create 的根本差异

task_create 建新 group，子任务拿到空白 fd 表/信号处置/环境/内存映射。
fork 复制当前 group，子任务拿到父 group 的一份副本——fd 表、信号处置表、环境变量、虚存映射都复制。

POSIX fork 语义的根源：fork 后父子进程共享文件偏移（fd 指向同一 file）、但信号处置是副本（改子不影响父）、环境是副本。NuttX 通过 task_fork.c 内复制 group 字段 + fdlist_copy 实现此语义（无独立 group_clone 函数，复制逻辑在 nxtask_setup_fork 内联完成）。

4.2 nxtask_setup_fork() 主链路（task_fork.c）

FAR struct tcb_s *nxtask_setup_fork(start_t retaddr)
{
  /* ① 分配子 TCB */
  child = kmm_zalloc(sizeof(struct tcb_s));

  /* ② 复制父 TCB 调度相关字段 */
  child->init_priority  = rtcb->init_priority;
  child->sched_priority = rtcb->sched_priority;
  child->start          = retaddr;   /* fork 返回地址 */
  child->entry          = rtcb->entry;

  /* ③ 复制 group 字段（task_fork.c 内联完成，无独立 group_clone 函数） */
  group_allocate(child, TCB_FLAG_TTYPE_TASK);   /* 暂建空 group */
  /* 此处 nxtask_setup_fork 内联复制 rtcb->group 的字段到 child->group */
  /* fdlist_copy 复制 fd 表（衔 VFS 文档 §7.2） */

  /* ④ 复制栈 */
  nxtask_setup_stackargs(child, NULL, rtcb->adj_stack_size);
  up_initial_forkstate(child);   /* arch 钩子：伪造子返回现场 R0=0 */

  /* ⑤ 不激活，等 nxtask_start_fork */
  return child;
}

up_initial_forkstate 是 fork 专用的 arch 钩子——伪造子的"返回现场"：子被首次切入时从 fork 调用点继续，R0=0（fork 返回 0 给子）；父的 R0=子的 pid（fork 返回子 pid 给父）。这是 fork 双返回的汇编根源。

4.3 nxtask_start_fork() 完成激活

pid_t nxtask_start_fork(FAR struct tcb_s *child)
{
  /* ① 分配新 pid */
  child->pid = nxtask_getpid();

  /* ② 入就绪 */
  nxtask_activate(child);

  /* ③ 父返回子的 pid */
  return child->pid;
}

激活后子等 PendSV 首次切入——从 up_initial_forkstate 伪造的 fork 调用点继续，R0=0。

4.4 fork 时各子系统的复制行为

子系统 fork 复制行为 POSIX 语义
fd 表fdlist_copy 复制父 fd 表到子 group fd 表，子 fd 指向父同一 file（refs+1）共享文件偏移
信号处置sigaction 表复制为副本（子改不影响父）disposition 是副本
信号 procmask复制子独立
环境变量tg_envp 复制为副本子独立
虚存映射tg_mm_map 复制（CONFIG_ARCH_ADDRENV 下）COW 语义（NuttX 简化为复制）
sem 持有fork 不复制父持有的 sem（衔信号量文档）子不持有父的 sem
mqueuefork 不复制父打开的 mqueue fd 的特殊状态标准 POSIX
timerfork 不复制 POSIX timer标准 POSIX

4.5 fork 失败回滚：nxtask_abort_fork()

void nxtask_abort_fork(FAR struct tcb_s *child, int errcode)
{
  /* 回滚 nxtask_setup_fork 已复制的 group 字段 */
  group_leave(child);   /* 摘 group */
  group_drop(child->group);   /* refs 减归零释放（sched/group/group_leave.c:231） */

  /* 回滚栈 */
  nxtask_uninit(child);   /* 释放栈、TCB */

  child->errcode = errcode;
}

fork 任何步骤失败（group 字段复制内存不足等）走 abort 回滚——回滚已复制的东西，避免半成品 group/tcb 残留。

5. 路径 ④：pthread —— 加入当前 group

pthread 是最简单的路径——不建新 group，只建新 tcb 加入当前 group：

pthread_create → nxtask_init(tcb, priority, entry, start, TCB_FLAG_TTYPE_PTHREAD)
  → group_allocate(tcb, TTYPE_PTHREAD)
       → 不建新 group，而是 tcb->group = this_task()->group
       → group_join(tcb)：把 tcb 挂到当前 group 的 tg_members
  → nxtask_setup_stackargs(tcb, NULL, stack_size)
  → nxtask_activate(tcb)

pthread 与 task_create 的唯一实质差异在 group_allocate 的 ttype 分支——PTHREAD 走 group_join 而非建新 group。这就是 pthread 共享 fd/信号处置/环境的实现根源。

pthread 的退出走 nxtask_exit → join_sem 唤醒 pthread_join 等者（衔 §6.3）。

6. 路径 ③：spawn 与 execve

6.1 posix_spawn() / task_spawn() 三步组合

spawn = task_init + spawn file_actions + execve：

  1. task_init 建新 task + 新 group（同路径①）。
  2. 应用 spawn file_actions 改 fd 表——posix_spawn_file_actions_t 描述一组 fd 操作（关/重开/dup2），在 fdlist_copy 后应用。
  3. execve 装载新代码镜像——替换当前 task 的 entry/start，fd 按 cloexec 关。

sched/task/task_spawn.c / task_posixspawn.c / task_spawnparms.c 实现此组合。

6.2 task_execve() 替换镜像（task_execve.c）

execve 不建新 task，只替换当前 task 的代码：

  1. 装载二进制：binfmt_load（binfmt/）解析 ELF/BIN，分配新代码段、设新 entry。
  2. 关 cloexec fd：遍历当前 fd 表，关所有 f_cloexec == true 的 fd——衔 VFS 文档 §7.3 的实质实现就在这里。
  3. 保留 signal disposition：execve 不清信号处置（除 SIGCHLD 等被设为 default）——POSIX 语义。
  4. 保留环境变量：execve 不清环境。
  5. 重置栈：分配新栈，up_initial_state 伪造现场指向新 entry。
  6. 重置 pid 不变：execve 后 pid 同前，只是代码换了。

execve 失败（装载错）返回原任务继续跑；成功则从新 entry 重跑——原代码后续不再执行。

6.3 spawn vs fork+execve

路径 步骤 优势
spawntask_init + file_actions + execve一步到位，无 fork 复制 group 的开销
fork+execvefork 复制 group → execve 替换镜像父进程可在 fork 后 execve 前改子环境（如关特定 fd）

NuttX 嵌入式场景常用 spawn——省 fork 的 group 复制开销，直接建新 task 装载。

7.4 exit 后的 Zombie 状态

exit 后 TCB 不立即释放——父未 wait 收尸时 TCB 保留在 zombie 状态：

  • task_state = TSTATE_TASK_INVALID
  • pid 还在 g_pid 表（可被查到）
  • 退出码存在父 group 的 tg_children 链
  • 栈和 TCB 内存不释放
Zombie 的意义：POSIX wait 必须能取到子退出码——TCB 必须留着等收尸。如果父不 wait，zombie 永驻（内存泄漏）。NuttX 有 system reaper 兜底机制（CONFIG_SCHED_HAVE_PARENT 下父先死则 reparent 到 init，init 回收）。

8. 退出路径 ②：task_delete / task_terminate —— 被外部强杀

8.1 task_delete()（task_delete.c）

int task_delete(pid_t pid)
{
  tcb = nxsched_get_tcb(pid);
  if (tcb == NULL) return -ECHILD;

  /* 不能删自己 */
  if (tcb == this_task()) return -EDEADLK;

  /* 设 KILLED 标志，等下次调度点切入时自杀 */
  tcb->flags |= TCB_FLAG_NONEX_STATUS_CANCEL;
  nxsig_notification(tcb, SIGKILL, SI_KERNEL, ...);  /* 投 SIGKILL */

  return OK;
}

task_delete 不立即杀——投 SIGKILL，目标在下个调度点切入时检测到 CANCEL 标志走 exit 路径自杀。这是 NuttX 的"延迟取消"设计：不能在别人上下文里乱释放资源，必须让对方自己有序退出。

8.2 task_terminate()（task_terminate.c）

更强——直接调 nxtask_exit：

int task_terminate(pid_t pid, int status)
{
  tcb = nxsched_get_tcb(pid);
  if (tcb == this_task()) return -EDEADLK;

  /* 标志后投 SIGKILL，与 delete 类似但能指定退出码 */
  tcb->flags |= TCB_FLAG_NONEX_STATUS_CANCEL;
  tcb->errcode = status;
  nxsig_notification(tcb, SIGKILL, SI_KERNEL, ...);
  return OK;
}

8.3 task_restart()（task_restart.c）

特殊——不杀，把 task 的 PC 重置回 entry，清栈，重新跑。CONFIG_SCHED_HAVE_PARENT 下子若 exit 还可 restart。这是 NuttX 嵌入式场景的"重启服务"机制。

8.4 cancelpt 取消点（task_cancelpt.c）

pthread_cancel 的取消机制——在取消点检查 CANCEL 标志，若置则走 exit。NuttX 把取消点埋在 syscall 边界（read/write/wait 等），pthread_cancel 设标志后目标在下次取消点自杀。

9.2 waitpid() 主链路

pid_t waitpid(pid_t pid, FAR int *statloc, int options)
{
  /* ① 在当前 group 的 tg_children 链找已退出的子 */
  child = group_find_child(group, pid);
  if (child != NULL && (child->ch_flags & CHILD_FLAG_EXITED))
    {
      /* 已退出：取码、摘链、释放 child_status_s、返回 pid */
      *statloc = child->ch_status;
      group_remove_child(group, child);
      group_free_child(child);
      return child->ch_pid;
    }

  /* ② 未退出：阻塞等 SIGCHLD */
  if (!(options & WNOHANG))
    {
      /* 等 SIGCHLD 投递唤醒——走 sigtimedwait 机制 */
      ret = nxsig_timedwait(SIGCHLD, &info, timeout);
      /* 唤醒后再查 tg_children */
      ...
    }
}

9.4 child_status_s 链的管理

子 exit 时 group_add_child 加节点；父 wait 时 group_remove_child 摘节点 group_free_child 释放。这就是 zombie 的生命周期：exit 建 zombie 节点 → wait 摘节点。

group_find_child 按 pid 查；group_exit_child 标 EXITED；group_remove_children 批量清（父 exit 时所有未收尸子一次性清）。

9.5 父先死：reparent 到 init

父先 exit 时子怎么办？POSIX 规定子 reparent 到 init（pid 1）——init 担任 reaper，定期 wait 收所有孤儿。

NuttX 实现（CONFIG_SCHED_HAVE_PARENT）：父 exit 时 group_remove_children 把所有未收尸子 reparent 到 init 的 tg_children 链。init 的 reaper 路径（sched_init_reaper 或 idle 兜底）定期 wait 收。

10.2 idle task（sched_idletask.c）

每核一个 idle task，最低优先级，无 group（kernel task）。空闲时跑——做 cpuload 统计、pm 低功耗、垃圾回收（如 mm 的延迟 free，衔内存管理文档 §6）。idle 永不退出。

14. 总结

  • 四种诞生路径：task_create（建新 group）、fork（复制 group）、spawn（建+装载）、pthread（加当前 group）。差异在 group_allocate 的 ttype 分支——建新 group vs 加入当前 group vs 复制当前 group。
  • group 是进程语义、tcb 是线程语义：fd 表/信号处置/环境/虚存映射在 group 上，同 group 的 tcb 共享（pthread）。fork 复制 group（fdlist_copy 共享文件偏移、信号处置副本）；task_create 建新 group（空白）；pthread 加入当前 group。
  • task_init 主链：kmm_zalloc TCB → group_allocate（按 ttype）→ group_initialize/bind → nxtask_setup_scheduler/stackargs/name → up_initial_state 伪造现场 → TSTATE_TASK_INACTIVE。nxtask_activate 入就绪等 PendSV 首次切入。
  • fork 双返回根源：up_initial_forkstate 伪造子返回现场 R0=0，父 R0=pid。子从 fork 调用点继续看起来跟正常切入一样。
  • exit 全链清理：wd_cancel waitdog、nxsig_cleanup 信号、mq_close mqueue、sem_destroy、fdlist_release 关所有 fd、timer_delete、env_release、join_sem post。建 child_status_s 节点存退出码、投 SIGCHLD、入 zombie。
  • wait 闭环：子 exit 投 SIGCHLD → 父 waitpid 阻塞等 SIGCHLD（走 sigtimedwait）→ 唤醒查 tg_children → 取码摘链释放 zombie。父先死 reparent 到 init。
  • task_delete 延迟杀：投 SIGKILL 设 CANCEL，目标在下个调度点自杀——不能在别人上下文里乱释放。
flowchart TD App[应用 API] --> P{诞生路径} P -->|task_create| TC[task_init 建新 group] P -->|fork| FK[task_fork 复制 group] P -->|spawn| SP[task_spawn 建+装载] P -->|pthread_create| PT[加当前 group] TC --> INIT[nxtask_init: kmm_zalloc TCB + group_allocate建新 + up_initial_state伪造] FK --> FORK[nxtask_setup_fork: 复制父group + up_initial_forkstate R0=0] SP --> SPAWN[task_init + file_actions + execve] PT --> PTHR[task_init TTYPE_PTHREAD group_join当前] INIT --> ACT[nxtask_activate 入就绪] FORK --> ACT SPAWN --> ACT PTHR --> ACT ACT --> Pendsv[PendSV 首次切入伪造现场 bx entry] Pendsv --> Run[任务跑起来] Run --> Ex{退出路径} Ex -->|exit/return| EXIT[nxtask_exit: exithook清理 + 建child_status + 投SIGCHLD + zombie] Ex -->|task_delete| DEL[投SIGKILL 设CANCEL 下调度点自杀] Ex -->|task_terminate| TERM[同DEL 但可指定退出码] EXIT --> Wait{父回收} Wait -->|wait/waitpid| W[阻塞等SIGCHLD 唤醒查tg_children 取码摘链] W --> Release[释放 zombie TCB + group refs减] EXIT --> Reparent{父先死?} Reparent -->|是| Init[reparent 到 init] Init --> InitWait[init reaper 定期wait收孤儿]
总结一句话：NuttX 任务生命周期 = 四种诞生（task_create/fork/spawn/pthread，差异在 group_allocate 的 ttype 分支）+ task_activate 入就绪 + 两种退出（自退出 nxtask_exit / 外部强杀 task_delete）+ wait/SIGCHLD 闭环回收 zombie。group 是进程语义（fd/信号处置/环境共享），tcb 是线程语义。exit 清理一切资源（wdog/信号/mqueue/sem/fd/timer/env），建 child_status 投 SIGCHLD，父 waitpid 取码释放。fork 复制 group（fdlist_copy），spawn 一步到位（task_init+execve），pthread 共享 group。
