NuttX 工作队列（work queue）详解

本文针对 NuttX 深入讲解工作队列子系统——sched/wqueue/ 1718 行五文件：内核异步执行设施，给下半层驱动、延迟执行、看门狗喂狗、通知派发用。

全部源码引用对应本仓库(nuttx/)里的真实文件，行号即真实行号。

与已写文档的衔面：

  • 与 nuttX_irq_management.md 衔——下半层驱动 ISR 唤醒 work thread 跑真正处理（threaded IRQ 类似）、ISR 不能阻塞的活推给 work_queue
  • 与 nuttX_watchdog.md 衔——CONFIG_WATCHDOG_AUTOMONITOR_BY_WORKER 用 work_queue 周期喂硬狗（看门狗文档 §5.2）
  • 与 nuttX_taskscheduling.md 衔——work thread 是普通 task（LPWORK/HPWORK），受调度器管，HPWORK 高优先级
  • 与 nuttX_task_create_exit.md 衔——work_usrstart 启动期建 work thread（os_bringup 后）
  • 与 nuttX_clock_timer.md 衔——work_queue 周期延迟执行靠 tick + wdog 倒计时
  • 与 nuttX_semaphore_mutex.md 衔——work queue 用 sem 同步唤醒

1. 概览：work queue 的角色

1.1 三类用户场景

场景 API 用途
下半层驱动work_queueISR 唤醒 work thread 跑真正处理（ISR 不能阻塞的活推给 wqueue）
延迟执行work_queue + delay周期/延迟执行内核工作（看门狗喂、统计更新、垃圾回收）
通知派发work_notifier系统事件（boot/resume/suspend）派发给注册回调

1.2 工作队列实例与优先级

NuttX 内建两类工作队列：

队列 配置 线程名 优先级 用途
LPWORKCONFIG_SCHED_LPWORKlpwork低优先级（CONFIG_SCHED_LPWORK_PRIORITY）不急的下下半层、周期喂狗、统计
HPWORKCONFIG_SCHED_HPWORKhpwork高优先级（CONFIG_SCHED_HPWORK_PRIORITY）紧急的下半层（需快速响应的 driver）

外加用户队列（CONFIG_SCHED_USRWORK）：应用可建自己的 work queue 实例（work_queue_create），指定名字/优先级/栈大小。

1.3 work vs ISR vs task

上下文 限制 用途
ISR不能阻塞、不能持 mutex、不能调 kmm 多数矮现场处理、置标志、唤 work
work thread普通任务上下文，可阻塞、可持 mutex、可调 kmm下半层、延迟活、复杂处理
应用 task完全用户态应用逻辑

work queue 是 ISR → 普通上下文的"延迟出口"——ISR 唤醒 work thread，work thread 跑真正处理。类似 Linux softirq/tasklet。

2. 核心数据结构

2.1 struct work_s（include/nuttx/wqueue.h:250）

struct work_s
{
  struct list_node node;     /* 挂到 work queue 链 */
  worker_t worker;           /* 处理函数 */
  FAR void *arg;             /* 透传给 worker 的参数 */
  clock_t qtime;             /* 入队时刻（tick） */
  clock_t delay;             /* 延迟 tick 数 */
};

关键设计：struct work_s 由调用方分配（常作为 driver 私有结构的一部分静态声明），work_queue 不分配——这避免 ISR 调 work_queue 时还要 kmm_alloc（ISR 不能调 kmm）。调用方持 work_s 实例，work_queue 把它挂到队列链。

2.2 struct kwork_wqueue_s（include/nuttx/wqueue.h:239 邻近）

struct kwork_wqueue_s
{
  pid_t pid;                 /* work thread pid */
  sem_t sem;                 /* 唤醒信号量（work thread 等队列非空） */
  struct list_node q;        /* 工作链头 */
  FAR struct work_s *start;  /* 链上最早到期的工作 */
  clock_t start_time;        /* 最早到期时刻 */
  /* 命名/优先级/栈等字段 */
};

每个 work queue 实例（LPWORK/HPWORK/用户）一个 kwork_wqueue_s，挂所有待执行 work 到 q 链。

2.3 worker_t 回调类型

typedef CODE void (*worker_t)(FAR void *arg);

worker 跑在 work thread 上下文，收 arg 参数。worker 可阻塞（持 mutex、调 sleep）、可调 kmm——比 ISR 自由。

3. work thread 主循环：kwork_thread.c

3.1 work thread 启动

work_usrstart() (wqueue.h:328) 在 OS bringup 后调——建 LPWORK/HPWORK/USRWORK thread：

int work_usrstart(void)
{
  /* 建 LPWORK thread（CONFIG_SCHED_LPWORK 启用） */
  kwork_thread_create(&g_lpwork, "lpwork", CONFIG_SCHED_LPWORK_PRIORITY, stack);

  /* 建 HPWORK thread */
  kwork_thread_create(&g_hpwork, "hpwork", CONFIG_SCHED_HPWORK_PRIORITY, stack);

  return OK;
}

work thread 是普通 task——kwork_thread_create 调 task_create 建一个 TCB，entry 是 kwork_thread 主循环。衔任务创建退出文档 §3.1。

3.2 kwork_thread 主循环

static int kwork_thread(FAR void *arg)
{
  FAR struct kwork_wqueue_s *wqueue = arg;

  for (;;)
    {
      /* ① 等最早到期 work */
      ret = kwork_thread_timeout(wqueue);   /* sem_timedwait 到 start_time */

      /* ② 取链上所有到期 work 跑 */
      nxmutex_lock(&wqueue->lock);
      while (!list_is_empty(&wqueue->q))
        {
          work = list_first_entry(&wqueue->q, struct work_s, node);
          if (work->qtime + work->delay > curr_tick) break;  /* 还没到期 */

          list_delete_fast(&work->node);
          worker = work->worker; arg = work->arg;
          work->worker = NULL;     /* 标记已摘 调用方可再用 */

          nxmutex_unlock(&wqueue->lock);
          worker(arg);             /* 跑 worker，可阻塞 */
          nxmutex_lock(&wqueue->lock);
        }
      /* 更新 start/start_time */
      nxmutex_unlock(&wqueue->lock);
    }
}

关键点：

  • sem_timedwait 等最早 work 到期——衔时钟文档 §4 sigtimedwait 底层走 wdog/hrtimer
  • 摘下 work 跑 worker 时解锁——worker 可能阻塞，不能持 wqueue 锁
  • worker 跑完再锁 wqueue 查下一个——一个 tick 可能跑多个到期 work
  • work thread 永不退出，for(;;) 循环

3.3 周期执行：worker 内再 work_queue

worker 跑完即从链摘下，要周期执行就在 worker 内调 work_queue_next 再挂自己：

static void my_periodic_worker(FAR void *arg)
{
  /* 跑活 */
  ...

  /* 再挂自己，10 tick 后再跑 */
  work_queue_next(LPWORK, &my_work, my_periodic_worker, arg, 10);
}

work_queue_next 与 work_queue 区别：work_queue_next 在 worker 内调，不重置 qtime（沿用当前 tick），避免延迟叠加。看门狗 automonitor BY_WORKER 喂狗就是此模式（衔看门狗文档 §5.3）。

4. API 集群

4.1 入队

API 作用
work_queue(qid, work, worker, arg, delay)挂 work 到 qid 队列，delay tick 后跑
work_queue_wq(wqueue, work, worker, arg, delay)指定 wqueue 实例（用户队列用）
work_queue_next(qid, work, worker, arg, delay)worker 内调，不重置 qtime
work_queue_priority(qid)取 qid 队列优先级

work_queue 主链（sched/wqueue/kwork_queue.c）：

  1. 检查 work->worker == NULL（空闲可再用），否则 -EAGAIN（已有 pending）
  2. 填 worker/arg/qtime = curr_tick/delay
  3. 锁 wqueue，挂到 q 链按到期时间升序插
  4. 更新 start/start_time 若更早
  5. sem_post(&wqueue->sem) 唤 work thread 重新算等待

4.2 取消

API 作用
work_cancel(qid, work)从 qid 队列摘 work
work_cancel_wq(wqueue, work)指定 wqueue 实例

work_cancel 锁 wqueue，从链摘 work，work->worker = NULL 标记空闲。摘后若 wqueue 空，sem_post 唤 thread 重等。

竞态：work_cancel 与 work thread 跑 worker 竞态——若 worker 已摘下还没跑，cancel 摘不掉（已不在链）。work_cancel 返回 0 表示成功摘，1 表示已摘下正在跑或已跑完。

4.3 通知派发：work_notifier

work_notifier_setup (wqueue.h:283) 注册系统事件回调：

  • WORK_BOOT —— 启动期
  • WORK_RESUME —— resume
  • WORK_SUSPEND —— suspend

事件触发时 work_notifier_signal 派发——把所有注册回调挂到 LPWORK 跑。常给 driver 做启动后初始化、suspend 前释放资源。

4.4 遍历：work_foreach

work_foreach(fn, arg) 遍所有活跃 work 调 fn——给调试/统计用。

4.5 用户队列：work_queue_create / work_queue_free

work_queue_create(name, priority, stack, flags) 建用户队列实例——分配 kwork_wqueue_s + 建 work thread。work_queue_free 反过来释放。应用/driver 可建自己专属队列避免与 LPWORK/HPWORK 争资源。

5. 与看门狗 automator BY_WORKER 衔接（衔看门狗文档 §5）

CONFIG_WATCHDOG_AUTOMONITOR_BY_WORKER 模式下，硬狗喂狗源是 work_queue：

  • watchdog_automonitor_worker worker 跑 lower->ops->keepalive 喂硬狗
  • worker 内调 work_queue_next(LPWORK, ..., WATCHDOG_AUTOMONITOR_PING_INTERVAL_TICK) 续自己
  • 若 work thread 饿死（LPWORK 优先级太低被高优先级抢占太久 / kmm 短 / 调度器卡），worker 跑不了 → 硬狗没人喂 → 复位 MCU

这是 work queue 监控活性的应用——衔看门狗文档 §5.2 六喂狗源。

6. 与 ISR/threaded IRQ 衔接（衔 IRQ 文档）

6.1 ISR 唤 work 的典型模式

static int my_isr(int irq, FAR void *arg, FAR void *regs)
{
  FAR struct my_dev_s *priv = arg;

  /* ISR 内只置标志 + 唤 work thread */
  priv->pending = true;
  work_queue(LPWORK, &priv->work, my_worker, priv, 0);   /* 0 delay 立跑 */

  return IRQ_HANDLED;
}

static void my_worker(FAR void *arg)
{
  FAR struct my_dev_s *priv = arg;

  /* work thread 上下文，可阻塞、可调 kmm */
  nxmutex_lock(&priv->lock);
  /* 跑真正的处理：读设备寄存器、转缓冲、唤用户 */
  ...
  priv->pending = false;
  nxmutex_unlock(&priv->lock);
}

ISR 不能阻塞的活（读设备寄存器常要等总线、转缓冲可能调 kmm）推给 work thread——这是 NuttX "ISR → 延迟出口" 模式。

6.2 与 threaded IRQ（CONFIG_IRQPTHREAD）的区别

threaded IRQ 是 NuttX 另一种"ISR → 普通上下文"机制（IRQ 文档 §2.4）——ISR 返 IRQ_WAKE_THREAD 唤专属 irq thread。work queue 是更通用的"延迟执行"设施，不限 IRQ；threaded IRQ 专给 ISR 用，更轻量但无延迟参数。

机制 用途 延迟 队列实例
work_queue通用延迟执行tick 延迟可选LPWORK/HPWORK/USRWORK
threaded IRQ专给 ISR0（立跑）每 IRQ 一个 irq thread

9. 总结

  • work queue 是 ISR → 普通上下文的延迟出口——ISR 不能阻塞的活（读设备、转缓冲、调 kmm）推给 work thread 跑。类似 Linux softirq/tasklet。
  • LPWORK/HPWORK/USRWORK 三类队列：低/高优先级内核队列 + 用户队列。work thread 是普通 task 受调度器管，HPWORK 高优先级需快速响应。
  • struct work_s 调用方分配——ISR 调 work_queue 不 kmm_alloc（ISR 不能调 kmm），调用方持 work 实例。
  • 主循 sem_timedwait 等最早到期——衔时钟文档 sigtimedwait 走 wdog/hrtimer。worker 跑完摘下，周期执行在 worker 内 work_queue_next 续自己（不重置 qtime 避免延迟叠加）。
  • 关端衔接：看门狗 BY_WORKER 用 work 周期喂硬狗（work thread 饿死→硬狗复位）；ISR 唤 work 跑下半层；通知派发挂回调到 LPWORK 跑。

小结一句话：NuttX 工作队列 = ISR → 普通上下文的延迟出口——struct work_s 由调用方分配（ISR 不调 kmm），挂到 kwork_wqueue_s.q 链按到期升序。work thread 永不退出，sem_timedwait 等最早到期后摘链跑 worker(arg)（可阻塞/持 mutex/调 kmm），摘下后 unlock 才跑 worker 避持锁阻塞。LPWORK/HPWORK/USRWORK 三类队列，周期执行在 worker 内 work_queue_next 续自己（不重置 qtime 避延迟叠加）。看门狗 BY_WORKER 用 work 周期喂硬狗（work thread 饿死→硬狗复位 MCU）。ISR 唤 work 跑下半层是 NuttX 驱动标准模式。
