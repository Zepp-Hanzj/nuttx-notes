NuttX 文件系统框架（VFS/inode）详解

本文针对 NuttX 深入讲解文件系统框架的统一骨架——VFS/inode/mount/fd 表四张表、driver 注册链路、open/close/read/write/ioctl 三大通用 VFS 路径、fcntl FD_CLOEXEC、fork/exec fd 继承关闭、pipe/socket/mqueue/epoll/eventfd/timerfd/signalfd 共用 fd 表、procfs 衔接。

全部源码引用对应本仓库（nuttx/）里的真实文件，行号即真实行号。

本文与已写文档的衔接：

  • 与 nuttX_mqueue_ipc.md 衔——mq_open 返回的 fd 与普通文件 fd 共用同一张 fd 表（§4）
  • 与 nuttX_taskscheduling.md 衔——fork/exec/fcntl FD_CLOEXEC、task 创建时 fd 继承（§7）
  • 与 nuttX_signal.md 衔——SIGPIPE/SIGIO 的投递、signalfd 用 fd 表（§7.5）
  • 与 nuttX_memory_mgmt.md 衔——inode/file/fd 的 kmm_alloc 分配（§3）
  • 与 nuttX_semaphore_mutex.md 衔——命名信号量走 inode（§3.4）
  • 与 nuttX_clock_timer.md 衔——timerfd 用 fd 表、fsync 在 reboot notifier（§7.4）

1. 概览：NuttX 文件系统框架的全景

1.1 VFS 的角色

NuttX 应用接口的所有 fd 类型——普通文件、设备驱动、管道、socket、mqueue、epoll、eventfd、timerfd、signalfd、inotify——共用一套骨架：fd 表（per task_group）→ file 表（全局）→ inode 表（全局树）→ inode 的 union inode_ops_u 联合体分流到四种操作集。

核心思想：fd 是 file 的索引，file 是 inode 的索引，inode 是 VFS 树的节点。fd/file 两表把"应用看到什么"和"内核怎么跑"解耦——应用只看 fd 和文件位置（f_pos），内核只看 inode 和操作集（i_ops/i_mops/i_bops）。

1.3 三种 inode 分流

union inode_ops_u (fs.h:380) 让同一个 inode 节点表达四种不同的东西，靠 i_flags 判定：

inode 类型 flags 判定 联合体字段 跑什么
driver（字符设备驱动）INODE_IS_DRIVERu.i_ops（file_operations）设备 open/read/write/ioctl
mountpoint（FS 挂载点）INODE_IS_MOUNTPTu.i_mops（mountpt_operations）FS open/stat/readdir/mkdir
block driver（块设备驱动）INODE_IS_BLOCKu.i_bops（block_operations）块 read/write(geometry)
MTD driverINODE_IS_MTDu.i_mtd（mtd_dev_s）MTD 设备

外加编译可选：

  • CONFIG_FS_NAMED_SEMAPHORES → u.i_nsem（命名信号量）
  • CONFIG_FS_NAMED_EVENTS → u.i_nevent（命名事件）
  • CONFIG_PSEUDOFS_SOFTLINKS → u.i_link（软链接路径）

file_operations vs mountpt_operations 是联合体：fs.h:224 注释明说二者前若干字段签名位置完全相同，被当 union 处理——这样 fs_close/fs_read/fs_write 不必区分 inode 是 driver 还是 mountpt，统一走 inode->u.i_ops->close / i_mops->close，编译期通过共用内存布局直接打通。

2. inode 表：VFS 伪文件树

2.1 struct inode 真身（fs.h:401）

struct inode
{
  FAR struct inode *i_parent;   /* 上一层节点（父） */
  FAR struct inode *i_peer;     /* 同层兄弟节点（链） */
  FAR struct inode *i_child;    /* 下层子节点（链） */
  atomic_t          i_crefs;    /* 引用计数 */
  uint16_t          i_flags;    /* INODE_IS_* 类型位 + 权限 */
  union inode_ops_u u;          /* 操作集联合体（四种分流） */
  ino_t             i_ino;      /* inode 序号 */
  size_t            i_size;     /* per inode driver 的私有数据大小 */
#ifdef CONFIG_PSEUDOFS_ATTRIBUTES
  mode_t            i_mode;     /* 权限位 */
  uid_t             i_owner;
  gid_t             i_group;
  struct timespec   i_atime;    /* 最近访问时间 */
  struct timespec   i_mtime;    /* 最近修改时间 */
  struct timespec   i_ctime;    /* 最近状态变更时间 */
#endif
  FAR void         *i_private;  /* per inode driver 私有数据指针 */
  char              i_name[1];  /* 节点名（变长，struct hack） */
};

i_parent/i_peer/i_child 三向链接构成一棵树+链表：

  • i_child 把同父的子节点串成链（首子）
  • i_peer 把兄弟节点串成链（同一父的兄弟）
  • i_parent 指父节点

查找路径 /dev/uart0 即：root → 找 dev 子节点 → 找 uart0 子节点。inode_find() (fs/inode/fs_inodefind.c) 实现此查找。

i_name[1] 是 C struct hack——inode_reserve() (fs/inode/fs_inodereserve.c) 分配时按名字长度多分字节，让名字嵌在结构末尾，一次 kmm_alloc 拿 inode + 名字，零碎片。

2.2 union inode_ops_u（fs.h:380）

union inode_ops_u
{
  FAR const struct file_operations    *i_ops;    /* driver */
#ifndef CONFIG_DISABLE_MOUNTPOINT
  FAR const struct block_operations   *i_bops;   /* block driver */
  FAR struct mtd_dev_s                *i_mtd;    /* MTD */
  FAR const struct mountpt_operations *i_mops;   /* mountpoint */
#endif
#ifdef CONFIG_FS_NAMED_SEMAPHORES
  FAR struct nsem_inode_s             *i_nsem;   /* 命名信号量 */
#endif
#ifdef CONFIG_FS_NAMED_EVENTS
  FAR struct nevent_inode_s           *i_nevent;
#endif
#ifdef CONFIG_PSEUDOFS_SOFTLINKS
  FAR char                            *i_link;   /* 软链接目标路径 */
#endif
};

四种主分流 + 三种可选分流共用同一块内存——一个 inode 时刻只能是一种。靠 i_flags 判定。INODE_IS_* 宏（fs.h 预处理段）按 flags 位判定。

2.3 全局根 g_root_inode（inode.h:166）

EXTERN FAR struct inode *g_root_inode;

整个 VFS 伪文件树只有一个全局根。inode_initialize() (fs/inode/fs_initialize.c) 启动时建根节点，所有路径查找从 g_root_inode 下行。

保护机制：fs/inode/fs_inode.c:50 定义 static rw_semaphore_t g_inode_lock = RWSEM_INITIALIZER;，inode_rlock()/inode_lock()/inode_unlock() (inode.h:191-213) 包 rw_semaphore：

  • 查找路径走读锁（多查找可并发）
  • 注册/注销/挂载/卸载写锁（独占）

2.4 struct inode_search_s（inode.h:134）—— 查找描述符

struct inode_search_s
{
  FAR const char *path;      /* INPUT: 待查路径；OUTPUT: 没走完的残余 */
  FAR struct inode *node;    /* OUTPUT: 找到的 inode */
  FAR struct inode *peer;    /* OUTPUT: 找到节点的左兄弟 */
  FAR struct inode *parent;  /* OUTPUT: 找到节点的父 */
  FAR const char *relpath;   /* OUTPUT: 若是 mountpoint，挂载点内的相对路径 */
  FAR char *buffer;          /* OUTPUT: 路径展开缓冲（软链接等） */
  bool nofollow;             /* INPUT: terminal 不解软链接 */
};

SETUP_SEARCH(d,p,n) 宏 (inode.h:50) 初始化；inode_find() / inode_search() 填返回字段；RELEASE_SEARCH(d) 宏 (inode.h:63) 释放 buffer。

relpath 字段的意义：当找到的 inode 是一个 mountpoint，相对路径（如 /mnt/data/log.txt 中的 log.txt）传给 FS 的 open(filep, relpath, ...)——挂载的 FS 只管自己挂载点内的相对路径，绝对路径前缀由 VFS 切掉。这是 NuttX mount 简化的体现。

2.5 inode 引用计数 i_crefs

atomic_t i_crefs 是原子引用计数：

  • 每次 file_vopen 成功找到 inode → inode_reserve 增一
  • 每次 file_close → inode_release 减一，归零则可释放

driver/block/MTD inode 通常是注册时长期存活的（ref 永不归零），unregister 才删。mountpoint inode 是 mount 时建、umount 时删。临时 inode（如 pipe 临时节点）ref 归零即释放。

3. file 表：打开文件实例

3.1 struct file 真身（fs.h:457）

struct file
{
  int               f_oflags;   /* open flags（O_RDONLY/O_RDWR/...） */
  atomic_t          f_refs;     /* 引用计数（dup 共享一 file） */
  off_t             f_pos;      /* 文件位置（read/write 共享） */
  FAR struct inode *f_inode;    /* 指向 inode */
  FAR void         *f_priv;     /* per file driver 私有数据指针 */
#if CONFIG_FS_LOCK_BUCKET_SIZE > 0
  bool              f_locked;   /* filelock 状态 */
#endif
};

关键设计：fd 表存的是 struct fd，struct fd 指向 struct file——多个 fd（dup 后）可指向同一 file，共享 f_pos 和 f_refs。这是 POSIX "dup 共享文件偏移"语义的根源。

f_priv 是 driver 私有数据——同一个 inode 被 open 多次，每次有自己的 f_priv。driver 的 open 通常分配自己的私有结构填进 f_priv，close 时释放。例：UART driver 的 f_priv 存波特率/数据位配置；pipe 的 f_priv 存 pipe 实例；socket 的 f_priv 存 socket 实例。

3.2 file 的分配/释放

fs/inode/fs_files.c：

  • file_allocate() (fs_files.c:779) —— 从全局池分配一个空 file，f_refs = 1
  • file_deallocate(filep) —— 归还池（f_refs 归零时）
  • file_addref(filep) / file_release(filep) —— 引用计数管理

file 表是全局的（不 per-group），跨 task_group 共享——同一文件被两个进程 dup 后，两个进程的 fd 指向同一全局 file，共享 f_pos。

3.3 fd 表：struct fd 与 struct fdlist（fs.h:469, 491）

struct fd
{
  FAR struct file  *f_file;     /* fd → file */
  bool              f_cloexec;  /* exec 时关闭此 fd */
#ifdef CONFIG_FDCHECK
  uint8_t           f_tag_fdcheck;  /* fdcheck 防 fd 错用 */
#endif
#ifdef CONFIG_FDSAN
  uint64_t          f_tag_fdsan;    /* fdsan 防 fd 误用 */
#endif
#if CONFIG_FS_BACKTRACE > 0
  FAR void         *f_backtrace[CONFIG_FS_BACKTRACE];  /* open 调栈 */
#endif
};
struct fdlist
{
  spinlock_t        fl_lock;    /* 保护 fd 表访问 */
  uint8_t           fl_rows;    /* fl_fds 数组的行数 */
  FAR struct fd   **fl_fds;     /* 两层 fd 数组（行 × 列） */
  FAR struct fd    *fl_prefd;   /* 预分配 fd，避免创建期调 kmm */
  struct fd         fl_prefds[CONFIG_NFILE_DESCRIPTORS_PER_BLOCK];
};

两层 fd 数组——fl_fds 是 struct fd * 的行指针数组，每行 CONFIG_NFILE_DESCRIPTORS_PER_BLOCK 个 fd。fl_rows 跟随 fd 用量扩张，避免一次分配巨型 fd 表。预分配 fl_prefds 保证 task 创建期（kmm 还没就绪时）也能开 stdin/stdout/stderr 三个 fd。

3.4 fd 表的归属：per task_group

struct task_group_s 有字段 tg_fdlist（sched/group/group.h）——每个 task_group 自己一张 fd 表。同 group 内的 task（如 pthread 同进程的多个线程）共享这张 fd 表——这是 POSIX 同进程线程共享 fd 的语义根源。

nxsched_get_fdlist_from_tcb(tcb) (fs/inode/fs_files.c) 从 tcb 取所属 group 的 fdlist——fs_open 等通用 VFS 路径都用它定位当前任务的 fd 表。

3.5 fd → file 的分配：fdlist_dupfile（fs_files.c:566）

int fdlist_dupfile(FAR struct fdlist *list, int oflags, int minfd,
                   FAR struct file *filep)

逻辑：

  1. 取 fdlist 写锁。
  2. 从 minfd 起找首个空 fd 槽位。
  3. 填 fd->f_file = filep、fd->f_cloexec = (oflags & O_CLOEXEC) != 0、FDCHECK/FDSAN/BACKTRACE tag。
  4. 返回 fd 编号。

O_CLOEXEC 在 fd 层而非 file 层——因为 dup 的两个 fd 可以一个 cloexec 一个不 cloexec（POSIX），共享的 file 不能背这个 flag，flag 必须挂在 fd 上。f_cloexec 字段在 struct fd 而非 struct file，正是为此。

4.2 file_vopen() 主链路（fs_open.c:74）

static int file_vopen(FAR struct file *filep, FAR const char *path,
                       int oflags, mode_t umask, va_list ap)
{
  struct inode_search_s desc;
  FAR struct inode *inode;

  /* ① O_CREAT 时取 mode 参数 */
  if ((oflags & O_CREAT) != 0)
      mode = va_arg(ap, mode_t);
  mode &= ~umask;

  /* ② inode 查找 */
  SETUP_SEARCH(&desc, path, (oflags & O_NOFOLLOW) != 0);
  ret = inode_find(&desc);
  if (ret < 0)
    {
#ifdef CONFIG_PSEUDOFS_FILE
      /* O_CREAT 且没找到 → pseudofile_create 建伪文件 inode */
      if ((oflags & O_CREAT) != 0)
          ret = pseudofile_create(&desc.node, path, mode);
#endif
      if (ret < 0) goto errout_with_search;
    }

  inode = desc.node;
  if (desc.nofollow && INODE_IS_SOFTLINK(inode))
      return -ELOOP;  /* O_NOFOLLOW 遇软链接报错 */

#if defined(CONFIG_BCH)
  /* ③ 块 driver 转 BCH 字符 driver 代理（递归再 open） */
  if (INODE_IS_BLOCK(inode) || INODE_IS_MTD(inode))
    {
      inode_release(inode);
      RELEASE_SEARCH(&desc);
      ret = block_proxy(filep, path, oflags);  /* 递归本函数 */
      return ret;
    }
#endif

  /* ④ 权限检查 */
  ret = inode_checkopenperm(inode, oflags);

  /* ⑤ file ← inode 关联 */
  filep->f_oflags = oflags;
  filep->f_inode  = inode;

  /* ⑥ 按 inode 类型分流调 open */
  if (oflags & O_DIRECTORY)
      ret = dir_allocate(filep, desc.relpath);          /* 目录 */
  else if (INODE_IS_MOUNTPT(inode))
      ret = inode->u.i_mops->open(filep, desc.relpath, oflags, mode);  /* FS */
  else if (INODE_IS_DRIVER(inode) || INODE_IS_PIPE(inode))
      ret = inode->u.i_ops->open(filep);                /* driver */
  else
      ret = -ENXIO;                                    /* 无对应 */

  /* ⑦ 失败回滚：inode release */
  if (ret < 0) goto errout_with_inode;

  RELEASE_SEARCH(&desc);
  return OK;

errout_with_inode:
  filep->f_inode = NULL;
  inode_release(inode);
errout_with_search:
  RELEASE_SEARCH(&desc);
  return ret;
}

要点：

  • ② inode_find 走 g_inode_lock 读锁下行树查找；找不到时若 O_CREAT + CONFIG_PSEUDOFS_FILE 走 pseudofile_create 建伪文件 inode。
  • ③ BCH 块转字符代理：块设备不能直接 read/write 字节流，BCH 子系统包一层字符 driver 代理——递归 block_proxy 再调本函数开代理字符 driver。这是 NuttX 给块设备提供流式接口的设施。
  • ⑥ 类型分流：mountpoint 调 i_mops->open 并传 relpath；driver/pipe 调 i_ops->open 不传 relpath。FS 与 driver 的 open 签名不同——这正是 file_operations 与 mountpt_operations 开放方法签名不同处。
  • ⑦ 回滚：任何失败回滚 inode 引用，file 不挂 inode——保证 file_close 时不会乱解。

4.3 nx_vopen() 分配 file（fs_open.c:270）

filep = file_allocate();           /* 全局池拿空 file */
ret = file_vopen(filep, ...);      /* 上面的主链路 */
fd = fdlist_dupfile(list, oflags, 0, filep);  /* 挂到当前 fd 表 */
if (fd < 0)
  { file_close(filep); file_deallocate(filep); }  /* fd 表满回滚 */

fd 表满（fdlist_dupfile 返回 -EMFILE）时回滚 file_vopen 已做的事——file_close 调 driver close + inode release，file_deallocate 还池。关键：file 表与 fd 表是两张独立表，file 表有可分配但 fd 表满时仍要回滚。

4.4 close 全链路：file_close()（fs_close.c:93）

int file_close(FAR struct file *filep)
{
  inode = filep->f_inode;

#ifdef CONFIG_FS_NOTIFY
  path = file_get_path(filep);     /* 先取路径，close 后 inode 没了 */
#endif

  if (inode)
    {
      file_closelk(filep);         /* 解 file lock */

      if (inode->u.i_ops && inode->u.i_ops->close)
          ret = inode->u.i_ops->close(filep);   /* 调 driver/mount close */

      if (ret >= 0)
        {
#ifdef CONFIG_FS_NOTIFY
          notify_close(path, filep->f_oflags);   /* inotify 通知 */
#endif
          inode_release(inode);                  /* inode 引用减一 */
        }

      filep->f_inode = NULL;
    }
  return ret;
}

注意 inode->u.i_ops——close 路径里直接走 i_ops，不区分 inode 是 driver 还是 mountpt。这是因为 file_operations 与 mountpt_operations 前 8 个方法签名位置完全相同（fs.h:224 注释），被当 union 共享——同一内存地址既是 i_ops->close 又是 i_mops->close，编译期地址同。

nx_close (fs_close.c:183) → file_close + fdlist_release（fd 表摘下，file release 减引用）。file release ≠ file close：dup 的两个 fd 共享一 file，一个 close 只 file_release 减 f_refs，不调 driver close；f_refs 归零才真正 close driver。

6. mount：用 inode 表达挂载点

6.1 mount 没有独立结构

nx_mount() (fs/mount/fs_mount.c:286) 的实质是给挂载点 inode 填 u.i_mops：

int nx_mount(FAR const char *source, FAR const char *target,
             FAR const char *filesystemtype, unsigned long mountflags,
             FAR const void *data)
{
  /* ① 找底层 driver（块或 MTD） */
  if (source != NULL)
    {
      if (find_blockdriver(source, mountflags, &drvr_inode) >= 0)
          mops = mount_findfs(g_bdfsmap, filesystemtype);  /* 块 FS */
      else if (find_mtddriver(source, &drvr_inode) >= 0)
          mops = mount_findfs(g_mdfsmap, filesystemtype);  /* MTD FS */
    }
  else
      mops = mount_findfs(g_nonbdfsmap, filesystemtype);   /* 无块 FS */

  /* ② 给 target inode 填 i_mops */
  SETUP_SEARCH(&desc, target, false);
  inode_find(&desc);
  mountpt_inode = desc.node;
  mountpt_inode->u.i_mops = mops;     /* ← 这就是 mount 的实质 */

  /* ③ 调 FS 的 bind 把底层 driver 接上 FS */
  mops->bind(drvr_inode, data, &fshandle);
  mountpt_inode->i_private = fshandle;
}

挂载完成后，target 路径的 inode 从 driver 类型变成 mountpoint 类型——再 open target 子路径就走 FS 路径而非 driver 路径。

mount_findfs() (fs_mount.c:251) 在三张全局 FS 表里找指定 filesystemtype：

  • g_bdfsmap —— 块设备 FS（FAT、smartfs、littlefs 等）
  • g_mdfsmap —— MTD 设备 FS（nxffs、spiffs 等）
  • g_nonbdfsmap —— 无块 FS（tmpfs、procfs、romfs、nfs 等）

7. fcntl / fork / exec 与 fd 表的协作

7.1 fcntl 全分支（fs/vfs/fs_fcntl.c）

cmd 行号 作用
F_DUPFD:62复制 fd，新 fd 清 cloexec
F_DUPFD_CLOEXEC:78复制 fd，新 fd 设 cloexec
F_GETFL:84取 file 的 f_oflags
F_SETFL(setfl)设 file 的 f_oflags（只能改 O_APPEND/O_NONBLOCK 等）
F_GETFD(getfd)取 fd 的 f_cloexec 位
F_SETFD:330设 fd 的 f_cloexec 位（只认 FD_CLOEXEC，fs_fcntl.c:338-345）

F_DUPFD / F_DUPFD_CLOEXEC 走 file_dup (fs/vfs/fs_dup.c)——同一 file 多挂一个 fd，file refs +1，新 fd 在指定最小值起的空槽位。

关键区分：

  • F_GETFD/F_SETFD 操作 fd 层的 f_cloexec（exec 时关闭）
  • F_GETFL/F_SETFL 操作 file 层的 f_oflags（O_APPEND 等打开模式）

fd 层标志只对这一个 fd 生效，dup 的兄弟 fd 不受影响；file 层标志所有共享该 file 的 fd 都生效（因为是同一个 file）。这是 POSIX 区分 fd flag 与 file flag 的根源。

F_SETFD 只接受 FD_CLOEXEC 一位（fs_fcntl.c:338 if (oflags & ~FD_CLOEXEC) return -EINVAL）——其他位都是保留的。

7.2 fork 时 fd 继承：group_setuptaskfiles（sched/group/group_setuptaskfiles.c:64）

int group_setuptaskfiles(FAR struct tcb_s *tcb,
                         FAR const posix_spawn_file_actions_t *actions,
                         bool cloexec)
{
  FAR struct task_group_s *group = tcb->group;

#ifndef CONFIG_FDCLONE_DISABLE
  /* 非 pthread 子任务复制父 group 的 fd 表 */
  if (group != rtcb->group &&
      (tcb->flags & TCB_FLAG_TTYPE_MASK) != TCB_FLAG_TTYPE_KERNEL)
    {
      ret = fdlist_copy(&rtcb->group->tg_fdlist,
                        &group->tg_fdlist, actions, cloexec);
    }
#endif
  /* 应用 spawn file_actions 改 fd */
  if (actions != NULL)
    { /* 关/重开/dup2 等操作 */ }
}

fork 继承 fd 的实质：fdlist_copy 把父 group 的 fd 表逐项复制到子 group 的 fd 表——子进程的每个 fd 重新指向父进程的同一 file（file refs +1），共享 f_pos。这是 POSIX fork 后父子进程共享文件偏移的根源。

cloexec 参数：fork 不带 cloexec（继承所有 fd）；posix_spawn / execve 带 cloexec=true，复制时跳过父 fd 表里 f_cloexec 的那些——这是 exec 关 fd 的实现。

kernel thread 不继承 fd——TCB_FLAG_TTYPE_KERNEL 跳过复制，内核线程不需要应用 fd。

7.3 exec 关 fd：FD_CLOEXEC 的运作

exec 时调 group_setuptaskfiles(tcb, actions, true)，fdlist_copy 带 cloexec=true：

  1. 复制父 fd 表到子 fd 表。
  2. 跳过父 fd 表里 fd->f_cloexec == true 的项——这些 fd 在新进程镜像里不存在，等于关闭。
  3. file 的 refs 不增——那些 cloexec 的 fd 在父进程也通常被关（除非父进程特意留）。

O_CLOEXEC 在 open 时设——fs_open.c fdlist_dupfile(list, oflags, 0, filep) 把 oflags & O_CLOEXEC 转到 fd->f_cloexec。open 时带 O_CLOEXEC 等价于事后 fcntl(fd, F_SETFD, FD_CLOEXEC)，省一次调用。

static int sync_reboot_handler(FAR struct notifier_block *nb,

7.6 procfs 衔点

fs/procfs/ 是 NuttX 的特殊 FS——mount 在 /proc，提供 /proc/<pid>/* 节点查 TCB 状态。procfs 的 mountpt_operations 是 g_procfs_operations (fs/procfs/)，read 时按 relpath 路径反查 TCB 字段——调调试接口出 tcb->pid/tcb->name/调度统计等。

8.2 伪文件系统（pseudofs）

CONFIG_PSEUDOFS_FILE 启用时，open 带 O_CREAT 找不到 inode 时 pseudofile_create (fs_open.c:111) 建一个伪文件 inode——存于内存 VFS 树，不持久。这是 NuttX 给应用用的"内存文件"设施，类似 tmpfs 但更轻。

伪文件 inode 的 i_ops 是 g_pseudofile_operations（fs/vfs/fs_pseudofile.c），read/write 走内存 buffer。unlink 调 i_ops->unlink（fs.h:248 #ifndef CONFIG_DISABLE_PSEUDOFS_OPERATIONS）释放。

8.3 软链接

CONFIG_PSEUDOFS_SOFTLINKS 启用时，inode 的 u.i_link 指向目标路径。inode_find 遍历路径遇软链接：

  • 中间节点遇软链接 → 展开 i_link 路径，继续下行（inode.h:120 注释）
  • 终节点遇软链接且 nofollow=false → 返回链接目标 inode
  • 终节点遇软链接且 nofollow=true（O_NOFOLLOW）→ 返回链接 inode 本身，file_vopen 检测到 INODE_IS_SOFTLINK 返回 -ELOOP（fs_open.c:131）

symlink / readlink API（fs/vfs/fs_symlink.c / fs_readlink.c）建/读软链接。

8.4 fs_heap：VFS 自有分配器

fs/fs_heap.c 提供 fs_heap_alloc / fs_heap_free——VFS 内部用的分配器，常基于 kmm。inode.h:68 RELEASE_SEARCH 释放 desc.buffer 时调 fs_heap_free。把 VFS 分配与通用 kmm 分离便于统计与限流。

  • 四张表：fd 表（per-group，两层索引）→ file 表（全局，f_pos/f_refs/f_inode）→ inode 表（全局树，g_root_inode）→ mount 表（无独立结构，用 inode 的 i_mops 表达）。fd → file → inode 三层解耦"应用看什么"与"内核怎么跑"。
  • 三种 inode 分流：driver（i_ops/file_operations）、mountpoint（i_mops/mountpt_operations）、block driver（i_bops/block_operations）、MTD（i_mtd）——靠 i_flags 判定。file_operations 与 mountpt_operations 前 8 方法签名位置相同被当 union 共享，read/write/ioctl/close 路径不显式区分 driver 还是 mount。
  • open 全链路：应用 open → nx_open → fdlist_open → nx_vopen → file_vopen → inode_find → 类型分流 → 调 driver/mount open → fdlist_dupfile 挂 fd。close 反过来：file_close → inode->u.i_ops->close → inode_release。
  • mount 无表：挂载的实质是给挂载点 inode 填 u.i_mops，从此该 inode 走 FS 路径。bind 接底层 driver，unbind 解绑。块/MTD driver 注册后才能被 mount 当 source。
  • fcntl FD_CLOEXEC：fd 层标志（per-fd）vs file 层标志（per-file 共享）。F_DUPFD/F_DUPFD_CLOEXEC 调 file_dup 共享 file 加 fd。F_SETFD 只认 FD_CLOEXEC 一位。
  • fork 继承 fd：fdlist_copy 把父 group fd 表复制到子 group，每个子 fd 指向父 file（file refs +1，共享 f_pos）。exec 关 fd：fork 带 cloexec=false 继承所有；exec 带 cloexec=true 跳过父 fd 表里 f_cloexec 的项——这是 exec 关 fd 的实现。kernel thread 不继承 fd。
  • 共用 fd 表的设施：pipe/socket/mqueue/epoll/eventfd/timerfd/signalfd/inotify 全部用同一张 fd 表（通过 file_allocate_from_inode），fd 是统一入口。timerfd 衔时钟、signalfd 衔信号、mqueue 衔 IPC。
  • 保护：inode 树 rw_semaphore（查找读锁并发，注册写锁独占）；file 持有即安全（open 时 inode reserve refs+1，操作期间 inode 不释放）；fd 表 spinlock。fsync 在 reboot notifier 自动刷盘。
  • 与调度/内存/信号/同步/IPC/时钟衔接：fd 表在 task_group（调度）；inode/file 由 kmm_alloc（内存）；SIGPIPE/SIGIO/signalfd（信号）；命名信号量走 inode（同步）；mqueue fd 共用 fd 表（IPC）；timerfd 共用 fd 表（时钟）。
小结一句话：NuttX VFS 的核心是四张表三层解耦——fd 表（per-group 两层索引）→ file 表（全局 f_pos/f_refs）→ inode 表（全局树 g_root_inode）→ mount 无独立结构用 inode 的 i_mops 表达。inode 按 i_flags 分流到 driver(i_ops)/mountpoint(i_mops)/block(i_bops)/MTD(i_mtd)，file_operations 与 mountpt_operations 前 8 方法签名相同被当 union 共享，read/write/ioctl/close 路径不区分。open 全链路：应用 → nx_open → fdlist_open → nx_vopen → file_allocate → file_vopen(inode_find + 类型分流 + driver/mount open) → fdlist_dupfile 挂 fd。fork 时 fdlist_copy 复制 fd 表共享 file（共享 f_pos），exec 时 cloexec=true 跳过 f_cloexec 的 fd。pipe/socket/mqueue/epoll/eventfd/timerfd/signalfd/inotify 全共用 fd 表。inode 树 rw_semaphore 保护（查找读锁并发，注册写锁独占），file 持有即安全（open 时 inode refs+1 不被释放），fd 表 spinlock。
