1. 概览：信号在 NuttX 里干什么

信号是 NuttX 里唯一能让任务"异步被打断"的机制——跨调度/IPC/时钟三大子系统：

信号用途 例子
异步通知事件ISR → 用户态任务"外设完了"（mq_notify走信号）
打断阻塞等sem_wait/mq_receive/read 阻塞中收信号 → 返回 EINTR
默认动作杀任务SIGKILL/SIGSEGV → 终止任务
定时唤醒alarm/sleep 走 SIGALRM
用户态互发kill(pid, sig) / sigqueue

1.1 与信号量/互斥锁的定位差异

维度 信号量 信号
触发源任何任务调 sem_postkill/sigqueue/ISR/定时器/mqueue 通知
目标等这 sem 的所有任务特定 PID/group 或进程组
携带数据无（仅 count）siginfo（signo/code/value 共三字段）
处理时机post 即唤醒等者延迟到目标任务自己上下文里处理（pend → deliver）
语义同步事件计数异步打断 + disposition 动作

一句话：信号量是"事件计数"，信号是"异步打断 + 在目标上下文里调 handler"。

2.2 三队列的关系

队列 TCB 字段 存什么 谁管
sigactionqtg_sigactionq（组级）该任务组注册的所有 sigaction 处理块nxsig_action注册时挂

sigpendactionq → sigpostedq 是单向流转——pend 等投递，deliver 时摘 pend 队头挂 posted，handler 返回后摘 posted 释放。

2.3 sigaction_s 处理块

include/nuttx/signal.h —— 每个 signo 的 disposition 注册：

struct sigaction_s
{
  sq_entry_t  entry;           /* 链表节点 */
  _sa_handler_t sighandler;    /* handler 函数指针(SIG_DFL/IGN/自定义) */
  sigset_t    sa_mask;         /* handler 跑期间额外屏蔽的信号 */
  int          sa_flags;       /* SA_RESETHAND 等 */
  uint8_t      signo;          /* 关联的信号号 */
};

SIG_DFL（默认动作，终止/忽略等）、SIG_IGN（忽略）、自定义函数指针——三 disposition 复用 sighandler 字段。

3. 投递全流程：kill → nxsig_deliver

3.1 nxsig_kill（sig_kill.c:44）

int nxsig_kill(pid_t pid, int signo)
{
  /* pid > 0 给特定任务；pid==0 给同组；pid==-1 给所有；pid< -1 给进程组 */
  stcb = nxsched_get_tcb(pid);   /* 找目标 TCB */
  if (stcb == NULL) return -ESRCH;
  return nxsig_dispatch(stcb, &info, false);   /* false=不走组投递 */
}

3.2 nxsig_dispatch（sig_dispatch.c:754）

nxsig_dispatch 是所有信号源（kill/sigqueue/timer/mqueue_notify/ISR）的统一入口：

int nxsig_dispatch(FAR struct tcb_s *stcb, FAR siginfo_t *info,
                    bool group_dispatch)
{
  if (group_dispatch)
    return nxsig_dispatch_group(stcb, info);  /* 走组广播 */
  else
    return nxsig_tcbdispatch(stcb, info, false);  /* 单任务 */
}

关键观察：

  • 未屏蔽信号不一定立即处理——若目标在跑（非阻塞），只 pend 到 sigpendactionq，等下个投递时机；
  • 投递时机靠 sigdeliverpend 标志驱动——arm_doirq 的 PendSV handler 检测此标志后调 up_schedule_sigaction 伪造栈框架（对应任务调度 §12）；
  • SIGKILL/SIGTERM 默认动作是同步执行的——不 pend，立即 task_delete，这是与 Linux 的关键差异（NuttX RTOS 不能等"下次切"才杀）。

关键设计：

  • 同任务不嵌套信号——TCB_FLAG_SIGNAL_ACTION 标志防 handler 里又投新信号被立即处理；新信号 pend 在 sigpendactionq，handler 返回后由 nxsig_unmask_pendingsignal 推下一轮；
  • sigprocmask 临时扩张——handler 跑期间屏蔽集 = 原 mask ∪ sa_mask ∪ {本 signo}，防 handler 被同类型信号打断；
  • PROTECTED/KERNEL 模式切用户态调——up_signal_dispatch 在架构层实现，切到用户态跑 handler，返回内核态；FLAT 模式直接调。

4. PendSV栈框架伪造：源码级展开

对应任务调度 §12 那张概念图，这里源码级展开投递是怎么"在目标任务上下文里"发生的。

4.1 为什么需要 PendSV

信号投递要求在目标任务自己的栈、目标任务自己的优先级上下文里跑 handler——不能在发送者的栈里调（发送者可能优先级不同、栈不同、甚至在 ISR 里）。PendSV 的"最低优先级、退完所有外设 IRQ 再进"特性给了一次干净的切入点。

4.2 触发链：arm_doirq → up_schedule_sigaction

arch/arm/src/armv7-m/arm_doirq.c line 89-97：

if (irq == NVIC_IRQ_PENDSV)
{
    if (tcb->sigdeliver)              /* sigdeliverpend 标志驱动 */
        up_schedule_sigaction(tcb);   /* 安装信号投递栈框架 */
    up_irq_save();
}

sigdeliver 字段是 nxsig_tcbdispatch 在挂 pend 时设的回调指针（值是 nxsig_deliver）。PendSV handler 检测到非 NULL 即调 up_schedule_sigaction。

5. 用户接口语义

5.1 sigaction(signo, act, oldact)（sig_action.c:142）

注册某信号的 disposition：

int sigaction(int signo, const struct sigaction *act, struct sigaction *oldact)
{
    /* 找/建 sigaction_s 节点挂入 tg_sigactionq */
    /* act->sa_handler = SIG_DFL / SIG_IGN / 自定义函数 */
    /* act->sa_mask = handler 跑期间额外屏蔽 */
    /* act->sa_flags = SA_RESETHAND(一次性) 等 */
}

5.2 sigprocmask（sig_procmask.c）

查/改 sigprocmask 屏蔽集：

how 行为
SIG_BLOCKmask |= set（追加屏蔽）
SIG_UNBLOCKmask &= ~set（解屏蔽）
SIG_SETMASKmask = set（直接设）

解屏蔽时调 nxsig_unmask_pendingsignal——查 sigpendactionq 里是否有新解屏蔽的信号，触发投递。

5.3 sigsuspend（sig_suspend.c）

临时换屏蔽集并等信号：mask = set; wait; 恢复原 mask。原子地"换屏蔽+等"避免窗口漏信号。

5.4 sigwait/sigtimedwait（sig_waitinfo.c/sig_timedwait.c）

等指定信号集合中任一信号——不调 handler，直接返收到的 signo。走 TSTATE_WAIT_SIG 阻塞，nxsig_tcbdispatch 检测 sigwaitmask 唤醒。

与 sigsuspend 的差异：sigsuspend 必走 handler；sigwait 不走 handler，直接取信号数据返回。

5.5 kill/sigqueue/tgkill（sig_kill.c/sig_queue.c/sig_tgkill.c）

6.2 mutex 不被打断的原因

信号量文档 §8.3 提过 nxsem_wait_irq 有守卫：

if (!(mutex && (errcode == EINTR || errcode == ECANCELED)))
    { /* 清理 */ }

mutex 等待被 EINTR/ECANCELED 打断时跳过清理，等者只被唤醒但留 holder 链里。理由：mutex 保护临界区，被打断后临界区数据半截不可用；mutex 让等者乖乖等到持有者解锁。应用要 abort mutex 等，只能 nxsem_reset 强推。

6.3 与 mqueue 的协作

mq_send/mq_receive 阻塞中收信号：会被打断返 EINTR（mqueue 不像 mutex 有"持者独占"约束，打断后只漏消息不坏数据）。走 nxmq_wait_irq 同构路径。

6.4 mq_notify 异步通知

mqueue 文档 §7 的 mq_notify 注册"空变非空时发信号"——nxsig_notification 调 nxsig_dispatch 把信号投给注册者，走本文 §3 投递流程。

7. 三 disposition 与默认动作

7.1 三 disposition

disposition 设置方式 行为
SIG_DFLsa_handler = SIG_DFL 或从未注册走默认动作（见 §7.2）
SIG_IGNsa_handler = SIG_IGN信号被丢弃，不入 pend 队，不唤 sigwait
自定义sa_handler = 函数指针调 handler（在目标上下文）

7.2 默认动作（sig_default.c）

信号 默认动作
SIGKILL/SIGTERM终止任务（同步调 task_delete）
SIGSEGV/SIGBUS/SIGFPE终止任务（异常信号）
SIGSTOP挂起任务（state → STOPPED，等 SIGCONT）
SIGCONT恢复挂起任务
其余（SIGCHLD 等）忽略

7.3 SIGKILL/SIGSTOP 不可屏蔽/不可改

SIGKILL（9）和 SIGSTOP（19）不能被 sigprocmask 屏蔽、不能被 sigaction 改 disposition——POSIX 规范保证"总能杀/总能停"。NuttX 在 nxsig_action/sigprocmask 入口校验返 EINVAL。

8. 可重入性与信号安全函数

8.1 handler 里能调什么

POSIX 规范把函数分两类：

  • async-signal-safe——handler 里可调，如 write/read/_exit/signal 等；
  • 非 safe——不可调，如 malloc/printf/sem_wait 等（可能持锁，handler 中调可能死锁）。

NuttX 在 include/signal.h 标注了哪些接口 async-signal-safe，handler 里应只调这些。

8.2 同任务不嵌套

nxsig_deliver 入口 TCB_FLAG_SIGNAL_ACTION 守卫——若已在投递中，新信号只 pend 不立即处理。这避免了 handler 里又收信号导致栈嵌套爆。

8.3 handler 改 sigprocmask 的保留

nxsig_deliver 恢复 sigprocmask 时不是简单还原，而是 XOR 算出"handler 主动改的位"，尽量保留这些改动。注释明说这逻辑不完美（REVISIT），某些边界会漏检测。

9. 与超时/看门狗的协作

9.1 sleep/nanosleep 走信号

sig_sleep.c/sig_nanosleep.c —— sleep/nanosleep 实现靠 SIGALRM 定时器或 hrtimer，到时投 SIGALRM 唤醒等者。被打断返 EINTR。

9.2 看门狗（wdog）

include/nuttx/wdog.h —— wd_start/wd_cancel 看门狗定时器，到时调回调。回调常是 nxsem_wait_irq/nxmq_wait_irq 给阻塞等者补 ETIMEDOUT——即看门狗不直接发信号，而是触发 IPC 子系统的超时回收路径。

9.3 sigtimedwait 的超时

sig_timedwait.c —— sigtimedwait 等信号集合但带超时，挂看门狗，到时 nxsig_wait_irq(stcb, ETIMEDOUT) 唤醒。
