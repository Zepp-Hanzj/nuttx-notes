NuttX 任务定义、调度与上下文切换详解

本文针对 NuttX（以 Cortex-M 的 armv7-m 架构为例）深入讲解：

  • 任务是怎么定义的（TCB）；
  • 调度器怎么组织就绪/阻塞队列；
  • 任务状态机的转换；
  • 触发抢占的软件中断 SVCall / PendSV 的作用与触发方式；
  • 一次任务切换和上下文切换的完整时序。

所有源码引用都对应本仓库（nuttx/）里的真实文件，行号即真实行号。

1. 任务的定义

1.1 TCB —— 任务控制块

每个任务/线程在内核里都对应一个 TCB（Task Control Block），定义在 nuttx/include/nuttx/sched.h（struct tcb_s，约 line 564 起）。

TCB 是 NuttX 任务调度的"心脏"，承载：

字段类 内容 例
调度元信息 优先级、调度策略、CPU 亲和、时间片 sched_priority, flags, cpu, timeslice
任务状态 当前状态、所属就绪/阻塞队列链 task_state, flink/blink
上下文 CPU 寄存器现场、栈 xcp.regs, stack_alloc_ptr, adj_stack_ptr
抢占锁 临界区计数 lockcount
PID/家庭关系 pid、父任务、所在 group pid, group
信号 待处理信号、投递队列、procmask sigprocmask, sigpendactionq

本文重点关注的字段：

struct tcb_s
{
  /* ... 上一段公共字段 ... */
  uint8_t  sched_priority;   /* 调度优先级 */
  int      lockcount;        /* sched_lock() 计数,非0=禁抢占 */
  volatile tstate_t task_state;  /* 见 §3 */
  FAR struct dq_entry_s *flink;  /* 双链队列前后指针 */
  FAR struct dq_entry_s *blink;
  /* ... */
#if !defined(CONFIG_BUILD_FLAT) || defined(CONFIG_ARCH_FPU)
  struct xcptcontext xcp;    /* CPU 现场:regs[] 指针、FPU 现场 */
#endif
  /* ... */
};

1.2 三类任务来源

来源 创建路径 备注
应用任务 task_create() / pthread_create() apps/ 里调 POSIX 接口，经 task_init.c→task_setup.c→task_activate.c 加入就绪队列
空闲任务 sched_idletask.c 每个 CPU 一个，最低优先级，无其它就绪任务时跑它，做 tickless、低功耗
内核服务任务 kmm、work 队列等 优先级高于应用，内核态运行

1.3 任务的"出生"：从创建到入就绪

apps/ 调 task_create(name, priority, stacksize, entry, arg) 后大致链路：

task_create() [apps/task_create.c]
task_init() [sched/task/task_init.c] — 分配 TCB、栈、group
task_setup() [sched/task/task_setup.c] — 设 argv/environ、入口
task_activate() [sched/task/task_activate.c] — 设置初始 CPU 现场、加入就绪队列
内部调 nxsched_add_readytorun() ← 关键调度入口
up_switch_context() — 若新任务比当前优先级高，触发抢占

task_activate() 设置的初始上下文是通过架构层 up_initial_state()（arch/arm/src/armv7-m/arm_initialstate.c）写入的 —— 它把任务的入口地址、栈、参数填进 TCB 的 xcp.regs，模拟"像被切出去过一次"的现场，后续第一次切换进它时就能像恢复一样"回到入口"。

2. 调度策略与优先级

NuttX 默认是基于优先级的可抢占调度，可选时间片轮转。

2.1 调度策略

策略 flags 标记 行为
FIFO（SCHED_FIFO） TCB_FLAG_SCHED_FIFO 同优先级 FIFO，跑到自愿阻塞才让位
RR（SCHED_RR） TCB_FLAG_SCHED_RR 同优先级轮转，见 sched_roundrobin.c，tick 中减 timeslice，到 0 让位
Sporadic（SCHED_SPORADIC） TCB_FLAG_SCHED_SPORADIC 周期性低优先级+补偿高优先级，见 sched_sporadic.c

2.2 优先级语义

  • 范围 0..255，值越大优先级越高；
  • sched_priority 是"当前调度优先级"，可能因 sporadic 而动态升降；
  • 空闲任务优先级 = 0，永远垫底。

2.3 抢占锁 / 临界区

抢占不是无条件的，内核用 lockcount 计数做软临界区：

sched_lock();      /* rtcb->lockcount++ —— 禁抢占 */
... critical region ...
sched_unlock();    /* lockcount--; 到0才真正放 pending 任务进来 */

实现见 sched/sched/sched_lock.c / sched_unlock.c。注意 sched_lock 不禁中断，只禁"被更高优先级任务抢出去"。

3. 任务状态机

nuttx/include/nuttx/sched.h line 244 起 enum tstate_e 定义了全部状态，注释清楚分成三大类：

3.1 状态分类

READY_TO_RUN 大类
PENDING / READYTORUN / ASSIGNED / RUNNING
BLOCKED 大类
INACTIVE / WAIT_SEM / WAIT_SIG / WAIT_MQ* / STOPPED
INVALID
TCB 未初始化

宏定义（line 281-286）把状态区间固化下来，调度器据此判断"该任务是否就绪"：

#define FIRST_READY_TO_RUN_STATE  TSTATE_TASK_READYTORUN
#define LAST_READY_TO_RUN_STATE   TSTATE_TASK_RUNNING     /* 含PENDING/ASSIGNED */
#define FIRST_BLOCKED_STATE       TSTATE_TASK_INACTIVE
#define LAST_BLOCKED_STATE        (NUM_TASK_STATES-1)

3.2 状态转换全图

stateDiagram-v2 [*] --> INVALID: TCB 分配 INVALID --> READYTORUN: task_activate() READYTORUN --> RUNNING: 调度选中 RUNNING --> READYTORUN: 被抢占/时间片到 RUNNING --> PENDING: sched_lock期间被挤占 PENDING --> RUNNING: sched_unlock RUNNING --> WAIT_SEM: sem_wait阻塞 RUNNING --> WAIT_SIG: sigwait/sleep RUNNING --> WAIT_MQNOTEMPTY: mq_receive队空 RUNNING --> WAIT_MQNOTFULL: mq_send队满 RUNNING --> TASK_STOPPED: SIGSTOP RUNNING --> INACTIVE: 未activate WAIT_SEM --> READYTORUN: sem_post唤醒 WAIT_SIG --> READYTORUN: 信号到来 WAIT_MQNOTEMPTY --> READYTORUN: 队列非空 WAIT_MQNOTFULL --> READYTORUN: 队列非满 TASK_STOPPED --> READYTORUN: SIGCONT INACTIVE --> READYTORUN: task_activate() RUNNING --> [*]: task_exit()

3.3 就绪大类内部三档详解

状态 含义 怎么进 怎么出
READYTORUN 在就绪队列里等，没被选中跑 nxsched_add_readytorun() 把刚就绪的任务加进去 调度选中 → RUNNING，或被更高优先级先跑就留着
PENDING READYTORUN→RUNNING 的瞬态过渡（见下文） nxsched_add_readytorun(): sched_lock 期间发现新就绪任务优先级高于当前，挂进 g_pendingtasks 队列，状态置 PENDING sched_unlock() 计数到 0，调 nxsched_merge_pending() 把 pendingtasks 并入 readytorun → 转 RUNNING 或 READYTORUN
ASSIGNED（仅 SMP） 多核里已选了 CPU、还没真正上核 nxsched_add_readytorun() SMP 路径中核间分派 该 CPU 取走 → RUNNING
RUNNING 正在该 CPU 上执行 调度选中 自愿阻塞 / 被更高优先级抢占 / 时间片到

3.4 阻塞大类

task_state 设为 TSTATE_WAIT_* 时，任务不在就绪队列，而在某个等待资源（sem、sig、mq……）的等待队列里。唤醒路径：

资源就绪（如 sem_post）
nxsched_remove_blocked(rtcb) — 从等待队列摘下
nxsched_add_readytorun(rtcb) — 加入就绪队列或 pending 队列
若优先级高于当前 RUNNING → up_switch_context() 触发抢占

阻塞状态枚举注释都标了"在等什么"：

状态 等什么
TSTATE_TASK_INACTIVE 已初始化但未 activate
TSTATE_WAIT_SEM 信号量 sem_wait
TSTATE_WAIT_SIG 信号 sigwait；也兼作 SLEEPING 的睡眠
TSTATE_WAIT_MQNOTEMPTY / MQNOTFULL POSIX 消息队列空/满
TSTATE_WAIT_PAGEFILL 缺页换入（legacy paging）
TSTATE_TASK_STOPPED SIGSTOP 后等 SIGCONT

4. 就绪/阻塞队列的组织

4.1 三条核心队列

队列 类型 存放 入队函数
就绪队列 g_readytorun 双链优先队列 所有 READYTORUN/RUNNING 任务 nxsched_add_prioritized() 按优先级插
pending 队列 g_pendingtasks 双链优先队列 sched_lock 期间"挤不掉当前任务"的高优先级任务 同上，sched_addreadytorun.c line 87
阻塞队列 各资源自己持有 WAIT_* 任务 nxsched_add_blocked()（sched_addblocked.c）

SMP 下还有 g_assignedtasks[cpu]，每核一份，放已分核但还没真上核的任务。

4.2 优先队列怎么排

nxsched_add_prioritized()（sched/sched/queue.h 内联）按 sched_priority 降序插入，队头永远是当前应跑的最高优先级任务。因此 RUNNING 任务也始终在队头，后面跟着 READYTORUN，PENDING 在另一条队。

// sched/sched/sched_addreadytorun.c 单核分支
if (nxsched_islocked_tcb(rtcb) &&
    rtcb->sched_priority < btcb->sched_priority)
{
    nxsched_add_prioritized(btcb, list_pendingtasks());
    btcb->task_state = TSTATE_TASK_PENDING;   /* §3.3 那个过渡态 */
    ret = false;
}
else if (nxsched_add_prioritized(btcb, list_readytorun()))
{
    btcb->task_state = TSTATE_TASK_RUNNING;
    btcb->flink->task_state = TSTATE_TASK_READYTORUN;
    up_update_task(btcb);
    ret = true;
}

5. 触发切换的软件中断：SVCall / PendSV

NuttX 在 Cortex-M 上用两个软件触发的异常来完成"在普通线程态里发起一次切换"：

异常 NVIC IRQ 怎么触发 用途
SVCall NVIC_IRQ_SVCALL（=11） 执行 svc 指令（带立即数） 用户态发起 syscall / 显式请求切换；见 arm_dispatch_syscall.S
PendSV NVIC_IRQ_PENDSV（=14） 写 NVIC INTCTRL 的 PENDSVSET 位 中断返回前发现"该切换了"，延后到所有外设 IRQ 退完后才切

5.1 为什么需要这两个软中断

Cortex-M 的中断硬件自动压栈 R0-R3、R12、LR、PC、xPSR（8 字），返回用 bx lr 配 EXC_RETURN。但任务切换需要换"整栈、整上下文"，而且：

  • 不能在外设 IRQ 处理函数中途切——会嵌套混乱；
  • 不能在普通线程态直接改 SP/LR 切换——栈指针换一半被抢就崩。

所以标准做法（整个 Cortex-M RTOS 都这样）：

  1. 在 IRQ 退栈前 pend 一个 PendSV（up_trigger_irq(NVIC_IRQ_PENDSV)），它优先级设成最低；
  2. 当前所有高优先级 IRQ 先退完；
  3. 最后 PendSV 取异常，在它里面换 SP、换 PSP、bx 出新任务现场——这就是切换点。

5.2 触发 PendSV 的代码

arch/arm/src/armv7-m/arm_trigger_irq.c —— up_trigger_irq(irq, cpuset)：

case NVIC_IRQ_PENDSV:
    pend_bit = NVIC_INTCTRL_PENDSVSET;   /* 写 INTCTRL 的 bit11 */
    break;
...
modifyreg32(NVIC_INTCTRL, 0, pend_bit);  /* 置位 NVIC->ICSR PendSVSET */

NVIC 看到 PENDSVSET 就把 PendSV 异常 pend，等优先级轮到它就进 handler。

5.3 中断退栈前触发 PendSV 的逻辑

arch/arm/src/armv7-m/arm_doirq.c —— arm_doirq(irq, regs) 是所有 IRQ 的中央处理。退栈前它会判断"该不该切"：

// arm_doirq.c line 53 (exception_direct 路径)
if (g_running_tasks[this_cpu()] != this_task())
{
    up_trigger_irq(NVIC_IRQ_PENDSV, 0);   /* 队头变了,pend一次切换 */
}

// arm_doirq.c line 82-104 PendSV 自己进来时
if (irq == NVIC_IRQ_PENDSV)
{
    /* 如果有信号要投递,在此处理(不嵌套其他 IRQ) */
    if (tcb->sigdeliver)
        up_schedule_sigaction(tcb);
    up_irq_save();   /* 关键: PendSV handler 内不再被打断 */
}
else
{
    irq_dispatch(irq, regs);   /* 普通外设 IRQ:调驱动回调 */
}

/* 任何 IRQ 退栈前都会更新调度信息 */
nxsched_switch_context(*running_task, tcb);
*running_task = tcb;
regs = tcb->xcp.regs;

也就是说 —— 每个 IRQ 结束前都会调 nxsched_switch_context，它只更新调度记账（栈溢出检查、sporadic、note），真正的"换寄存器"动作是在汇编 exception_common 末尾根据 tcb->xcp.regs 取现场返回时完成的。

5.4 SVCall 的用途

SVCall 主要干两件事（见 arm_svcall.c line 95-105）：

switch (cmd)  /* R0 在 svc 调用时带的命令号 */
{
    case SYS_restore_context:
    case SYS_switch_context:
        tcb = this_task();
        restore_critical_section(tcb, this_cpu());  /* 换到新任务现场 */
        break;
    case SYS_syscall_return:        /* 用户态 syscall 返回 */
    case SYS_call_weak:             /* 弱符号 */
    ... /* 其它 syscall 命令 */
}

up_switch_context(tcb, rtcb)（API 在 nuttx/include/nuttx/arch.h line 515）的实现在架构层 arm/src/armv7-m/ 里最终会执行一条 svc SYS_switch_context，主动把自己 SVCall 进 handler，handler 内调 restore_critical_section 把栈换成目标任务的现场后返回 —— 即一次同步切换。

svc 与 IRQ 的区别在于：它是线程态主动发起的（像系统调用），所以能在 sched_unlock、sem_post、yield 这种"我在线程态、现在就想让位"的地方用；而 PendSV 是中断末尾被动 pend 的。

6. 一次任务切换的完整时序

设场景：任务 A 在跑，任务 B 在等信号量，某 IRQ 里 sem_post 把 B 唤醒，B 优先级高于 A。

T0 外设 IRQ 触发，硬件压 R0-R3/R12/LR/PC/xPSR，进 exception_common（arm_exception.S）
T1 exception_common 调 arm_doirq(irq, regs)
T2 arm_doirq → irq_dispatch → 驱动 ISR → 内部 sem_post()
T3 sem_post 调 nxsched_remove_blocked(B) — B 摘离 sem 等待队列
T4 sem_post 调 nxsched_add_readytorun(B)
B.sched_priority > A.sched_priority，且 A 未 sched_lock → B 进就绪队列队头，task_state = RUNNING → A（原队头）→ READYTORUN
T5 arm_doirq 调 nxsched_switch_context(A, B) — 仅做记账，不现场切
T6 arm_doirq 退，exception_common 取 tcb->xcp.regs = B 的现场指针
T7 exception_common ldmia 恢复 B 的 R4-R12，msr PSP = B 的 SP
T8 exception_common bx EXC_RETURN 退异常 → 退到 B 上次被切的点继续跑
关键：第 T6-T8 步是切换真正发生的瞬间。因为 Cortex-M 异常返回的 bx EXC_RETURN 会自动从 PSP/MSP 弹出硬件压的 8 字 PC/xPSR 等，所以前面把 PSP 换成 B 的 SP、把 R4-R12 换成 B 的寄存器后，这一退就退进了 B。

6.1 sched_lock 期间的延迟切换

若上面 T4 时 A 正在 sched_lock()（A->lockcount > 0），nxsched_add_readytorun 走另一分支：

// sched_addreadytorun.c line 80
if (nxsched_islocked_tcb(rtcb) &&
    rtcb->sched_priority < btcb->sched_priority)
{
    nxsched_add_prioritized(btcb, list_pendingtasks());
    btcb->task_state = TSTATE_TASK_PENDING;   /* ← 见 §3.3 PENDING */
    ret = false;
}

B 进 pending 队列，状态置 TSTATE_TASK_PENDING，不真切换。等 A 调 sched_unlock() 把 lockcount 减到 0：

// sched_unlock.c line 97
ptcb = dq_peek(list_pendingtasks());
if (ptcb && nxsched_merge_pending())          /* pending 并入就绪队列 */
{
    up_switch_context(this_task(), rtcb);     /* ← 同步 svc 切换 */
}

nxsched_merge_pending()（sched_mergepending.c）把 pending 队逐个并入就绪队列，队头若变化则触发 up_switch_context —— 这一步是 svc 软中断，同步切到 B。

7. 上下文切换的汇编详解

7.1 Cortex-M 现场组成

一次完整的任务现场 = 硬件自动压栈的 8 字 + 软件补充压的 R4-R11（及 FPU 的 S16-S31）。exception_common（arch/arm/src/armv7-m/arm_exception.S）统一处理。

硬件自动压栈（HW_XCPT_SIZE, 8字）
xPSR / PC(R15) / LR(R14) / R12 / R3 / R2 / R1 / R0
软件补压（SW_XCPT_SIZE）
BASEPRI / CONTROL / EXC_RETURN / R11 / R10 / ... / R4
← xcp.regs 指向此
FPU 寄存器（仅 CONFIG_ARCH_FPU）
FPSCR / S31 / ... / S16
信号投递栈框架（若 sigdeliver）
伪造的 sigdeliver 调用栈

TCB 的 xcp.regs 就指向"软件补压段"的顶端 —— 因为硬件段会随异常自动弹，只有软件段需要保存在 TCB 里跨切。

7.2 exception_common 入栈（切出方）

arm_exception.S line 124-172，精简版：

exception_common:
    mrs  r0, ipsr            /* r0 = 异常号 */
    mrs  r12, control

    /* EXC_RETURN(R14) bit2 告诉我们现场在 MSP 还是 PSP */
    tst  r14, #EXC_RETURN_PROCESS_STACK
    beq  1f
    mrs  r1, psp             /* 用 PSP */
    b    2f
1:  mrs  r1, msp             /* 用 MSP,且要给它留空间 */
    sub  r2, r1, #SW_XCPT_SIZE
    msr  msp, r2
    isb
2:
    /* 保存 R4-R11 + EXC_RETURN + SP/PRIMASK */
    stmdb r1!, {r2-r12,r14}

    /* bl arm_doirq —— 见 §5.3,在此可能换 tcb->xcp.regs */
    bl   arm_doirq

    /* 返回时 r0 = 新任务的 xcp.regs(可能是另一个任务的现场) */
    ldmia r0!, {r2-r12,r14}  /* 恢复新任务的 R4-R12 等 */

    /* 用 EXC_RETURN 选择退到 PSP 还是 MSP */
    tst  r14, #EXC_RETURN_PROCESS_STACK
    msreq msp, r0
    msrne psp, r0            /* 通常任务的现场在 PSP */

    bx   r14                 /* bx EXC_RETURN,硬件自动弹 R0-R3/R12/LR/PC/xPSR */
关键理解：这一段汇编同时服务"切出方压栈"和"切入方弹栈"。arm_doirq 返回值 r0 决定了切到哪个任务的现场 —— 若没有切换，r0 还是原任务的 regs，原地弹回；若有切换，r0 指向新任务的 regs，后续 bx 就退进了新任务。

7.3 切入方的第一次恢复

新任务如果是刚 activate 的、还没跑过，它的 xcp.regs 是 up_initial_state()（arm_initialstate.c）伪造的现场 —— 入口 PC、清零的 R4-R11、栈顶 LR 设成 task_exit 路径（防止任务函数 return 漏走）。第一次被切进去时，ldmia 恢复这些伪造值，bx EXC_RETURN 就退到任务 entry，跟正常切入看起来一模一样。

7.4 FPU 现场

CONFIG_ARCH_FPU 时，line 151-170 处理 S16-S31（非易失浮点寄存器）：

tst  r14, #EXC_RETURN_STD_CONTEXT   /* 硬件已自动压 S0-S15? */
ite  eq
vstmdbeq r1!, {s16-s31}    /* 没自动压则我们压 S16-S31 */
subne r1, #(4*SW_FPU_REGS) /* 自动压了则跳过这段 */

硬件 lazy 压栈通过 EXC_RETURN.FType 位决定，这里跟它对齐。

8. sched_unlock merge pending 实例

把 §6.1 那条路径完整展开。设场景：

A 在跑，sched_lock() 计数 2
IRQ 里 sem_post 唤醒 B（优先级高于 A）
nxsched_add_readytorun(B)：A->lockcount>0 且 B>A → B 进 g_pendingtasks 队头 → B->task_state = TSTATE_TASK_PENDING
IRQ 退，arm_doirq 检查队头：队头还是 A，不切 → 回到 A 继续跑
A 调 sched_unlock() 第一次 → lockcount=1，不动
A 调 sched_unlock() 第二次 → lockcount=0，走出口路径
/* sched_unlock.c line 97 */
irqstate_t flags = enter_critical_section_notrace();
ptcb = dq_peek(list_pendingtasks());   /* 拿到 B */
if (ptcb && nxsched_merge_pending())   /* 把 pending 全并入就绪队列 */
{
    up_switch_context(this_task(), rtcb);  /* svc 同步切到 B */
}
leave_critical_section_notrace(flags);

nxsched_merge_pending()（sched_mergepending.c）做的事：

  1. 遍历 g_pendingtasks，每个 TCB 用 nxsched_add_prioritized 按 sched_priority 插入 g_readytorun；
  2. 若有任务插到就绪队头（比当前 RUNNING 还高优先级），返回 true，且队头任务 task_state 置 RUNNING、原队头置 READYTORUN；
  3. 队头不变则返回 false，不切。

返回 true → up_switch_context 触发 svc → 进 SVCall handler → restore_critical_section 把 tcb->xcp.regs 换成新队头任务的现场 → 退异常时切到新任务。

8.1 restore_critical_section 干什么

restore_critical_section(tcb, cpu) 在架构层实现（armv7-m 里通常在 arm_svcall.c 关联或 arm_internal.h），核心：

/* 把当前 CPU 的"临界区状态"切换到新任务的版本 */
restore_critical_section(tcb, this_cpu());
/* 之后 regs = tcb->xcp.regs —— exception_common 末尾就退到新任务 */

每个任务的"临界区状态" = BASEPRI / PRIMASK / CONTROL 的值，这些在切出时一并压在 xcp.regs 里（SW_XCPT_SIZE 段），切入时一并恢复，保证新任务不会"意外继承了原任务的关中断态"。

9. SMP 与多核派发

CONFIG_SMP 时，§4-§6 几乎都走另一条分支。

9.1 队列结构差异

维度 单核 SMP
就绪队列 一份 g_readytorun 一份 g_readytorun（共享）+ 每核一份 g_assignedtasks[cpu]
pending 队列 g_pendingtasks 不用，直接 IRQ 给目标核
队头判定 队头即 RUNNING 各核 RUNNING 在 g_assignedtasks[cpu]，就绪队列队头只是"等核的"

9.2 任务派发 nxsched_deliver_task

nxsched_switch_running（sched_addreadytorun.c line 152）做的事：挑就绪队列里"亲和允许、且优先级高于当前核任务"的任务，把它抬到当前核的 g_assignedtasks[cpu]，并给当前核 pend 一次 PendSV。

9.3 跨核 PendSV

up_trigger_irq(irq, cpuset) 第二参数是 cpuset，实现里：

if (irq >= NVIC_IRQ_FIRST)
    putreg32(irq - NVIC_IRQ_FIRST, NVIC_STIR);   /* STIR 可指定目标核 */

NVIC STIR 寄存器（Software Trigger Interrupt Register）的高位写目标 CPU 集合，低位写 IRQ 号 —— 跨核 pend 中断，目标核被中断后做切换。

10. 调度锁与延迟抢占的"压栈效应"

sched_lock 是可嵌套的（计数 lockcount），嵌套期间所有"应抢占"事件都 pend。这导致一个压栈效应：

sched_lock() → lockcount=1
sem_post 唤醒 B（prio_B > prio_A）→ B 入 pendingtasks
sched_lock() → lockcount=2
sem_post 唤醒 C（prio_C > prio_B）→ C 入 pendingtasks，pending 队头现在是 C
sched_unlock() → lockcount=1，不解锁，nothing
sched_unlock() → lockcount=0
nxsched_merge_pending()：把 pendingtasks 全并入就绪队列（B 和 C），队头若是 C，切到 C；否则切到队头
up_switch_context → svc → 切到 C

pending 队列里优先级严格降序——nxsched_add_prioritized 每次插时按优先级找位置。所以哪怕唤醒顺序乱，解锁时并入就绪队列的队头也总是当时最高优先级那个。

10.1 sched_lock 与中断的关系

场景 IRQ 抢占
sched_lock() 计数 >0 IRQ 可进 IRQ 内能改就绪队列，但不能切（队头若变，只 pend PendSV；PendSV 自己在 handler 内 up_irq_save，直到 unlock 才真正切）
sched_lock() == 0 IRQ 可进 IRQ 末尾 arm_doirq 直接把队头切过去（exception_common bx 退到新任务）
up_irq_save()（关 IRQ） IRQ 不进 不存在抢占问题

所以 sched_lock 不禁中断、只禁"被更高优先级任务抢出去"。这跟 up_irq_save 不同 —— 后者是硬件层关 IRQ，前者是调度层"我先不让位"。

11. 队头抢占判定规则汇总

谁会触发 up_switch_context / pend PendSV？NuttX 里分散多个调用点，共性是：就绪队头变了，且新队头优先级 > 当前 RUNNING。主要触发点：

触发源 文件 切换方式
sched_unlock 解 pending sched_unlock.c:101 up_switch_context svc 同步切
时间片到（RR） sched_roundrobin.c:86 up_switch_context
优先级被改（sched_setpriority） sched_setpriority.c:97/131/179 up_switch_context
sleep/nanosleep 醒 sched_sleep.c:125/174 up_switch_context
sched_suspend 挂自己 sched_suspend.c:209 up_switch_context
IRQ 内唤醒高优先级 arm_doirq.c:55 up_trigger_irq(PENDSV) pend
sched_yield 让位 sched_yield.c up_switch_context

切换判定都走同一个函数 nxsched_add_readytorun，它返回 true 时才真切。返回 true 的条件：新任务插到了就绪队头（优先级严格大于原队头，或等优先级 + RR 时间片到）。

12. 与信号投递的协作

信号投递也是一次"软中断式的切换"，但比普通抢占复杂——要在"目标任务的上下文里"调信号 handler。NuttX 的做法：

  1. 标记待投递：sigdeliver 字段非 NULL 表示该任务有信号待投递；
  2. pend PendSV：在 arm_doirq 里 PendSV handler 自身处理（line 89-97）：
if (irq == NVIC_IRQ_PENDSV)
{
    if (tcb->sigdeliver)
        up_schedule_sigaction(tcb);   /* 安装信号投递栈框架 */
    up_irq_save();
}
  1. up_schedule_sigaction（arm_schedulesigaction.c）在目标任务的栈上伪造一个"调 sigdeliver 函数然后 restore 原 LR"的栈帧，让目标任务下次被切入时先跳进信号投递函数，投递完再回到原本断点。

之所以用 PendSV 而不是直接切 —— 信号投递要求在目标任务自己的栈、目标任务自己的优先级上下文里跑 handler。PendSV 的"最低优先级、退完所有外设 IRQ 再进"特性正好给了一次干净的切入点。

小结一句话：NuttX 任务调度 = 优先级队列 + 调度锁延迟抢占 + Cortex-M 异常机制软切换。 就绪队列队头永远是要跑的那个，切换分两条路——线程态主动 svc 同步切（up_switch_context）、中断末尾 pend PendSV 延迟切（arm_doirq+up_trigger_irq）；真切的瞬间在 exception_common 汇编末尾 bx EXC_RETURN，靠换 tcb->xcp.regs 和 PSP/SP 完成。 状态机的 PENDING 就是"锁期间已就绪、等解锁切"的过渡档，blocked 是在等具体资源、不在就绪队列的另一大类。
