NuttX procfs 与 tmpfs 详解

本文针对 NuttX 深入讲解两个特殊内存文件系统：

  • procfs（fs/procfs/，8076 行 15 文件）：mount 在 /proc，提供 /proc/<pid>/*、/proc/meminfo、/proc/cpuinfo、/proc/uptime、/proc/version 等节点实时生成内核状态信息——VFS 给"伪文件"做调试接口的典型用法，read 时按 relpath 路径反查 TCB/heap/cpuload 字段。
  • tmpfs（fs/tmpfs/fs_tmpfs.c，3139 行单文件）：mount 在 /tmp 等，提供真文件读写但内容存内存（kmm/umm 分配）——非持久、断电即丢，常作应用临时文件区。

全部源码引用对应本仓库(nuttx/)里的真实文件，行号即真实行号。

与已写文档的衔面：

  • 与 nuttX_vfs_inode.md 衔——procfs/tmpfs 都用 mountpt_operations 操作集（VFS 文档 §3.2），挂载点 inode 的 u.i_mops 指向它们；relpath 路径切割（VFS 文档 §4.2）
  • 与 nuttX_taskscheduling.md / nuttX_task_create_exit.md 衔——/proc/<pid>/* 节点反查 TCB 字段、/proc/<pid>/status 出 pid/name/state/priority
  • 与 nuttX_memory_mgmt.md 衔——/proc/meminfo 出 mallinfo 统计、/proc/<pid>/statm 出任务内存占用、tmpfs 文件块用 kmm/umm 分配
  • 与 nuttX_clock_timer.md 衔——/proc/uptime 出系统启动 tick 数、/proc/<pid>/sched 出调度统计
  • 与 nuttX_irq_management.md 衔——/proc/interrupts 出 IRQ 注册表（CONFIG_PROCFS_INTERRUPTS）

1. 概览：两个内存 FS 的角色

维度 procfs tmpfs
实现位置fs/procfs/ 15 文件 8076 行fs/tmpfs/fs_tmpfs.c 单文件 3139 行
mount 路径/proc（典型）/tmp 等（任意）
文件内容实时生成——read 时按 relpath 反查内核字段真存内存——write 写入块，read 读出块
持久性无（每次 read 实时算）无（断电即丢）
可写性大多只读，少数节点可写（profile 等）完全可读可写
操作集g_procfs_operations（procfs_operations 特化版）g_tmpfs_operations（标准 mountpt_operations）
用途调试接口、内核状态自省应用临时文件区、IPC 缓冲、配置中转

关键区分：procfs 是"伪文件"——readdir 列出节点名，但 read 时才按 relpath 算内容返回；不持久。tmpfs 是"真文件"——write 写入内存块，read 读出，跟普通 FS 一样只是后端是内存。

2. procfs：伪文件实时生成机制

2.1 procfs_operations 操作集（include/nuttx/fs/procfs.h:41）

struct procfs_operations
{
  /* 与 mountpt_operations 前 6 个方法签名位置相同（union 共享） */
  CODE int     (*open)(FAR struct file *filep, FAR const char *relpath,
                       int oflags, mode_t mode);
  CODE int     (*close)(FAR struct file *filep);
  CODE ssize_t (*read)(FAR struct file *filep, FAR char *buffer,
                       size_t buflen);
  CODE ssize_t (*write)(FAR struct file *filep, FAR const char *buffer,
                        size_t buflen);
  CODE int     (*poll)(FAR struct file *filep, FAR struct pollfd *fds,
                       bool setup);

  /* procfs 特化方法 */
  CODE int     (*dup)(FAR const struct file *oldp, FAR struct file *newp);
  CODE int     (*opendir)(FAR const char *relpath, FAR struct fs_dirent_s **dir);
  CODE int     (*closedir)(FAR struct fs_dirent_s *dir);
  CODE int     (*readdir)(FAR struct fs_dirent_s *dir, FAR struct dirent *entry);
  CODE int     (*rewinddir)(FAR struct fs_dirent_s *dir);
  CODE int     (*stat)(FAR const char *relpath, FAR struct stat *buf);
};

关键：procfs 用独立的 procfs_operations 而非标准 mountpt_operations——open 签名带 relpath/oflags/mode（与 mountpt 同），但 opendir/readdir 等 procfs 特化方法不与 mountpt 共享地址（VFS 文档 §3.2 mountpt_operations 也有这些方法但位置不同）。所以 procfs 不能与普通 mountpt 共用 union——它有自己的入口分发逻辑。

2.2 procfs_entry_s 入口表（procfs.h:103）

struct procfs_entry_s
{
  FAR const char *name;                    /* 节点名如 "meminfo" */
  FAR const struct procfs_operations *ops; /* 该节点的操作集 */
};

procfs 的根目录有一组静态入口表，列出 /proc 下的固定节点：

  • version —— fs_procfsversion.c，出 NuttX 版本字符串
  • uptime —— fs_procfsuptime.c，出 g_tick_started 起的 tick 数（衔时钟文档）
  • meminfo —— fs_procfsmeminfo.c，出 mallinfo 统计（衔内存文档）
  • cpuinfo —— fs_procfscpuinfo.c，出 CPU 个数/架构
  • cpuload —— fs_procfscpuload.c，出每核 busy/idle 百分比（衔时钟文档）
  • pressure —— fs_procfspressure.c，出 PSI 压力指标
  • critmon —— fs_procfscritmon.c，出 critical section 时长监控（衔 IRQ 文档 §11）
  • fdt —— fs_procfsfdt.c，出 Flatten Device Tree dump
  • iobinfo —— fs_procfsiobinfo.c，出 IOB 缓存统计
  • profile —— fs_procfsprofile.c，可写——控制 profiling 开关
  • interrupts ——（CONFIG_PROCFS_INTERRUPTS）出 IRQ 注册表 g_irqmap（衔 IRQ 文档 §2.2）

每个 entry 有自己的 procfs_operations 实例，open/read/stat 各自实现。

2.3 /proc/<pid>/* 子目录：fs_procfsproc.c（fs/procfs/fs_procfsproc.c）

/proc 下还有一类动态节点——/proc/<pid> 目录，每个活跃 task 一个。fs_procfsproc.c 的 opendir 遍历 g_pid 表所有活跃 tcb 生成 <pid> 子目录项；<pid> 子目录下又有固定节点：

节点 出什么 衔
statuspid/name/state/priority/flags调度文档 §1 + 任务创建退出 §2
statm任务内存占用（stack_size/heap）内存文档
sched调度统计（切换次数/cpuload）时钟文档
cmdlineargv 拼接任务创建退出 §3.3
cwd当前工作目录VFS
exeentry 函数地址任务创建退出 §3.3
stack栈 dump（CONFIG_PROCFS_TASK_STACK）内存文档

fs_procfsproc.c 的 readdir 遍 g_pid 列出所有 <pid>；open("/proc/123/status") 解析 relpath="123/status" → 找 pid=123 的 tcb → 找 "status" 子节点 → 调该节点的 open。

2.4 procfs_file_s 与 procfs_dir_priv_s（procfs.h:115, 122）

struct procfs_file_s   /* 每个 open 文件实例 */
{
  FAR const struct procfs_entry_s *procfsentry;  /* 指向入口表项 */
  /* 子类可加自己的字段 */
};

struct procfs_dir_priv_s   /* 每个 open 目录实例 */
{
  struct fs_dirent_s dir;                        /* VFS 目录结构 */
  FAR const struct procfs_entry_s *procfsentry;  /* 指向入口表项 */
  /* 子类可加遍历状态字段 */
};

procfs 的 open/opendir 分配这两个结构填到 filep->f_priv（VFS 文档 §3.1 file 的私有数据指针），read/readdir 从 f_priv 取状态做遍历。

2.5 read 实时生成内容

典型 read 实现模式（fs_procfsversion.c 为例）：

static ssize_t version_read(FAR struct file *filep, FAR char *buffer,
                            size_t buflen)
{
  ssize_t total = 0;
  FAR const char *p = g_version_string;   /* 静态版本字符串 */
  while (total < buflen && *p != '\0')
      buffer[total++] = *p++;
  return total;   /* 返回读出字节数 */
}

实时生成：每次 read 都重新遍 g_version_string 拷到 buffer——不在内存里存。/proc/<pid>/status 的 read 遍 tcb->name/tcb->pid/tcb->task_state/tcb->sched_priority 等字段格式化输出（衔调度文档 §1.1 TCB 字段）。

无 f_pos 概念：多数 procfs 节点 read 不用 filep->f_pos——每次 read 从头算，_seek 不影响。但 <pid>/stack dump 类节点用 f_pos 做分页读。

2.6 fs_skeleton.c 模板

fs/procfs/fs_skeleton.c 是给开发者加新 procfs 节点的模板——复制改 name/open/read/stat 四方法即可。新节点要在 procfs 根入口表加一行 procfs_entry_s。

3. tmpfs：真文件存内存

3.1 mountpt_operations 操作集（fs_tmpfs.c）

tmpfs 用标准 mountpt_operations（VFS 文档 §3.2）——完整支持 open/close/read/write/mkdir/readdir/unlink/rename/stat 等，跟普通 FS 一样。后端用 struct tmpfs_file_s（内存块链）存文件内容。

3.2 tmpfs 数据结构（fs_tmpfs.c 内部）

struct tmpfs_file_s     /* 一个文件 */
{
  struct tmpfs_file_s *tflink;     /* 父目录的子文件链 */
  mode_t tf_mode;                  /* 权限位 + 类型（文件/目录） */
  size_t tf_size;                  /* 文件当前大小 */
  FAR struct tmpfs_data_s *tf_data; /* 内容块链头 */
  char tf_name[1];                 /* 变长名 hack（struct hack） */
};

struct tmpfs_data_s     /* 一段内容块 */
{
  FAR struct tmpfs_data_s *dlink;  /* 块链 */
  size_t datlen;                   /* 本块有效字节 */
  char dat[1];                     /* 变长数据 hack */
};

文件内容用 tmpfs_data_s 块链存——每次 write 增/扩块，每次 read 按 f_pos 找块读出。这是 tmpfs 的"真文件"实现：跟普通 FS 一样按 f_pos 随机读写，只是块在 kmm/umm 分配而非盘。

3.3 mount 与 inode

mount("/tmp", "tmpfs", ...) → g_nonbdfsmap 找到 tmpfs 的 mountpt_operations（VFS 文档 §6.1）→ 给挂载点 inode 填 u.i_mops → bind 调 tmpfs_bind 建根 tmpfs_file_s 目录节点 → 之后 open("/tmp/foo") 走 VFS 文档 §4 open 链路，relpath="foo" 传给 tmpfs_open，在根目录子链找/建 "foo" 节点。

3.4 内存来源与配额

CONFIG_FS_TMPFS_MALLOC_LIMIT（默认无限）限制 tmpfs 总分配量——防止应用写爆 kmm 导致内核崩。配额计在每个 tmpfs 实例的 tg_alloc_size 字段，每次 write 增块前检查超配额即返回 -ENOSPC。

3.5 write 增长与 truncate

write 超过当前 tf_size 增/扩块链——tmpfs_data_s 块大小常 CONFIG_FS_TMPFS_BLOCK_SIZE（默认 512）。truncate 截短 tf_size 并释放多余块。

6. 总结

  • procfs 是"伪文件实时生成"——read 时按 relpath 反查内核字段（TCB/heap/cpuload/IRQ 注册表）格式化输出，不在内存存。用独立 procfs_operations 而非标准 mountpt_operations。fs_skeleton.c 给加新节点模板。
  • tmpfs 是"真文件存内存"——write 写入 tmpfs_data_s 块链（kmm/umm 分配），read 按 f_pos 取块。完整支持 open/close/read/write/mkdir/readdir/unlink/rename。CONFIG_FS_TMPFS_MALLOC_LIMIT 限总量防 kmm 爆。
  • 根本差异：procfs 不持久（每次算）、几乎不占内存、大多只读；tmpfs 持久（mount 期间）、按文件大小占内存、完全可写。
  • VFS 衔接：两者都用 mountpt 机制挂载——挂载点 inode 的 u.i_mops 指向它们的操作集，open 链路切 relpath 给 FS 内部处理（VFS 文档 §4.2）。
  • 调试接口：procfs 是 NuttX 给"伪文件做调试接口"的典型——/proc/<pid>/status、/proc/meminfo、/proc/cpuinfo、/proc/interrupts 出内核状态供应用/脚本自省。tmpfs 是"临时文件区"——应用临时配置/IPC 缓冲/中转文件。
flowchart TD Mount["mount /proc procfs"] --> ProcfsBind["procfs_bind 建根入口表"] Mount2["mount /tmp tmpfs"] --> TmpfsBind["tmpfs_bind 建根目录 tmpfs_file_s"] ProcfsBind --> ProcfsOp["procfs_operations 特化操作集"] TmpfsBind --> TmpfsOp["标准 mountpt_operations"] Open1["open /proc/meminfo"] --> Rel1["VFS 切 relpath=meminfo"] Open2["open /proc/123/status"] --> Rel2["VFS 切 relpath=123/status"] Open3["open /tmp/foo"] --> Rel3["VFS 切 relpath=foo"] Rel1 --> ProcfsOpen["procfs_open 找入口表 meminfo"] Rel2 --> ProcfsOpen2["procfs_open 解 123 找 tcb 再找 status"] Rel3 --> TmpfsOpen["tmpfs_open 在根子链找/建 foo"] ProcfsOpen --> Read1["read 实时算 mallinfo 格式化输出"] ProcfsOpen2 --> Read2["read 遍 tcb 字段格式化"] TmpfsOpen --> TmpfsWrite["write 增 tmpfs_data_s 块到 tf_data 链"] TmpfsWrite --> TmpfsRead["read 按 f_pos 找块读出"]
小结一句话：procfs 是"伪文件实时生成"——procfs_operations 特化操作集，read 时按 relpath 反查内核字段（TCB/heap/cpuload/IRQ 注册表）格式化输出；/proc/<pid> 子目录遍 g_pid 表找 tcb。fs_skeleton.c 给加新节点模板。tmpfs 是"真文件存内存"——标准 mountpt_operations，write 写入 tmpfs_data_s 块链（kmm/umm 分配），read 按 f_pos 取块。CONFIG_FS_TMPFS_MALLOC_LIMIT 限总量防 kmm 爆。两者都通过 mountpt 机制挂载，挂载点 inode 的 u.i_mops 指向各自操作集。
