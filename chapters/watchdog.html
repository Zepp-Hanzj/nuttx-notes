NuttX 看门狗子系统详解

本文针对 NuttX 深入讲解两套名字都叫 watchdog/wdog 但本质完全不同的机制：

  • 软件看门狗（wdog）（sched/wdog/）：按 tick/hrtimer 排序的软件定时器链表，到期在中断上下文调回调函数，不复位任何东西。POSIX timer、信号超时、mqueue 超时、mutex 超时都靠它。
  • 硬件看门狗（watchdog）（drivers/timers/watchdog.c + 各 arch 下半层）：驱动片上硬件看门狗外设，默认超时硬件复位 MCU。可用 WDIOC_CAPTURE 改成调用户回调，或用 CONFIG_WATCHDOG_AUTOMONITOR_BY_* 让内核用另一个独立定时源替你喂狗。
  • 软狗与硬狗在 CONFIG_WATCHDOG_AUTOMONITOR_BY_WDOG 模式下联动：软狗到期喂硬狗，软狗调度卡死→硬狗没人喂→硬狗复位 MCU。
  • 与调度、信号、IPC、时钟、panic dump 的接合。

全部源码引用对应本仓库（nuttx/）里的真实文件，行号即真实行号。

注意：本文与 nuttX_clock_timer.md §4 互补。该文已介绍软狗在"时钟三层"中的位置与基础 API，本文重点覆盖硬件看门狗子系统、软狗源码细节订正、两套看门狗的超时行为对比、automonitor 六种喂狗源、panic notifier、capture 改复位为回调等该文未覆盖的内容。涉及软狗基础概念处会引用该文，不重复展开。

1. 全景：两套看门狗的对比

1.1 命名陷阱

NuttX 源码里"watchdog"和"wdog"两个词被混用，但分属两套独立子系统。阅读源码时必须先看路径才能判断在讲哪一个：

实际指代 常见符号
软件看门狗struct wdog_s、wd_start、wd_cancel、wd_timer、g_wdactivelist、wdentry_t
硬件看门狗struct watchdog_lowerhalf_s、watchdog_ops_s、watchdog_register、WDIOC_*、/dev/watchdog0

两个名字都带"watchdog"是因为软件看门狗在 NuttX 历史上最早就是当作"轻量看门狗"设计的——到期调一个回调函数"提醒"系统。但它的行为根本不符合"看门狗"通常的语义（超时复位），实际只用于内核内部延迟执行。只有硬件看门狗才真正具备"超时复位 MCU"的能力。

2.2 struct wdog_s 真实定义（include/nuttx/wdog.h:85）

struct wdog_s
{
  struct list_node node;    /* 挂到 g_wdactivelist 的双向链表节点 */
  wdparm_t         arg;     /* 回调函数的参数 */
  wdentry_t        func;    /* 到期要执行的函数; NULL 表示非活跃 */
#ifdef CONFIG_PIC
  FAR void        *picbase; /* PIC 基址, 回调时 up_setpicbase 用 */
#endif
  clock_t          expired; /* 到期的绝对 tick 时间 */
};

活跃性判定：wdog->func == NULL 即非活跃，宏 WDOG_ISACTIVE(wdog) 即此判断。wd_start 设 func、wd_cancel 与 wd_expiration 摘下时清 func = NULL，这就是软狗的"启停位"。

2.3 g_wdactivelist 链表组织（sched/wdog/wd_initialize.c:42）

struct list_node g_wdactivelist = LIST_INITIAL_VALUE(g_wdactivelist);

所有活跃软狗按 expired 升序挂在此链表上——最早到期的在表头。这是经典有序链表组织（而非 hash wheel 或 RB-Tree），优点是插入/摘头都是 O(n) 但常数极小、表头查询 O(1)；缺点是大量软狗时插入退化 O(n)。NuttX 软狗典型同时活跃数 < 100，此结构足够。

2.5 超时回调链路：wd_timer → wd_expiration

底层 tick 中断或 hrtimer 到期 → wd_timer()（wd_start.c:358） → wd_expiration()（wd_start.c:107）：

static inline_function clock_t wd_expiration(clock_t ticks)
{
  flags = enter_critical_section();
  wd_update_expire(ticks);          /* tickless: 记本次到期基准 */
  wd_set_nested(true);              /* 标记"正在回调中" */

  while (!list_is_empty(&g_wdactivelist))
    {
      wdog = list_first_entry(&g_wdactivelist, struct wdog_s, node);
      if (!clock_compare(wdog->expired, ticks))   /* 表头还没到期, 跳出 */
        {
          next_ticks = wdog->expired;
          break;
        }
      list_delete_fast(&wdog->node);  /* 摘下 */
      func = wdog->func; arg = wdog->arg;
      wdog->func = NULL;              /* 标记非活跃 */
      up_setpicbase(wdog->picbase);   /* PIC 基址修正 */
      CALL_FUNC(func, arg);           /* 调用户回调 */
    }

  wd_set_nested(false);
  if (next_ticks != ticks)
    wd_timer_start(next_ticks, true); /* 重设底层 timer 对准新表头 */
  leave_critical_section(flags);
  return next_ticks;
}

要点：

  • while 循环会一直摘链表头并调回调，直到表头 expired 还没到当前 tick——一次 tick 中断可能触发多个软狗回调（几个软狗恰在同一 tick 到期时）。
  • 回调运行在 tick 中断或 hrtimer 中断上下文里，受所有 ISR 限制：不能调阻塞 API、不能持 mutex、不能做长操作。
  • wd_set_nested(true) 关键——防止回调里用户 wd_cancel 别的软狗时去重设底层 timer；此时底层 timer 已经在跑这个中断，再 start 会乱套。
  • up_setpicbase(wdog->picbase) 修正 PIC（位置无关代码）基址，保证回调跑在 wd_start 调用时的地址环境里——NuttX 对 PIC 内核的特殊处理。
  • 软狗不复位任何东西——它只是在中断上下文里调你给的回调函数，这就是软狗"超时"的全部后果。

2.6 取消：wd_cancel()（wd_cancel.c:61）

int wd_cancel(FAR struct wdog_s *wdog)
{
  flags = enter_critical_section();
  if (WDOG_ISACTIVE(wdog))
    {
      first = list_first_entry(&g_wdactivelist, struct wdog_s, node);
      list_delete_fast(&wdog->node);
      wdog->func = NULL;
      if (first == wdog && !wd_in_callback())
        {
          /* 被取消的恰是链表头且不在回调中: 重设底层 timer */
          if (!list_is_empty(&g_wdactivelist))
            wd_timer_start(wd_next_expire(), false);
          else
            wd_timer_cancel();
        }
      ret = OK;
    }
  leave_critical_section(flags);
  sched_note_wdog(NOTE_WDOG_CANCEL, ...);
  return ret;
}

关键点：只有被取消的是链表头且不在回调中（!wd_in_callback()）才需要重设底层 timer——因为底层 timer 是按"旧表头的到期时刻"设的，旧表头没了就要改对准新表头。非头节点取消不影响底层 timer。

2.7 软狗的三种底层驱动模式（wdog.h:107-195）

通过 CONFIG_HRTIMER / CONFIG_SCHED_TICKLESS 编译开关切换三种底层 tick 源：

模式 触发 wd_timer 签名 重设底层方式
普通 tick每个 tick 中断void wd_timer(clock_t ticks)（隐式 1 tick）无需重设，tick 自然前进
ticklessup_timer_tick_start 设的下一闹钟void wd_timer(clock_t ticks)wd_timer_start 调 up_timer_tick_start(next_tick - clock_systime_ticks())
hrtimer高精度定时器到期uint64_t wd_timer(const hrtimer_t *timer, uint64_t expired)hrtimer_start(&g_wdtimer, wd_timer, TICK2NSEC(tick), HRTIMER_MODE_ABS)

wd_adjust_next_tick()（wdog.h:143）还处理：

  • CONFIG_SCHED_TICKLESS_LIMIT_MAX_SLEEP——不让系统睡超过 g_oneshot_maxticks，避免长睡期间错过外部 IRQ。
  • CONFIG_TIMER_ADJUST_USEC——减去中断延迟的最坏情况补偿，让 tickless 闹钟更准（wdog.h:154-168 注释详述）。

2.8 软狗的真实用途（非看门狗）

软狗不是"看门狗"，而是 NuttX 内核的通用延迟回调机制。典型用户：

用户 回调 补什么
POSIX timer（timer_create）timer_wdog_handler（drivers/timers/timer_wdog.c:244）到期发信号给 owner
信号超时回收（sigtimedwait）nxsig_timedwaitnxsig_wait_irq(tcb, ETIMEDOUT)
mqueue 发/收超时nxmq_sndtimeout / nxmq_rcvtimeoutnxmq_wait_irq(tcb, ETIMEDOUT)
信号量超时回收nxsem_rcvtimeoutnxsem_wait_irq(tcb, ETIMEDOUT)
mutex 持有超时nxmutex_timeout唤醒等者补 ETIMEDOUT
work_queue 延迟（部分配置）work_trampoline延迟执行工作
应用直接调 wd_start用户回调用户自定义（仍受 ISR 限制）

软狗的本质是"延迟 + 中断上下文回调"，与"看门狗"通常的"超时复位"语义无关。称它"看门狗"是 NuttX 历史命名习惯。

3. 硬件看门狗（watchdog）上下半层结构

3.1 上半层 / 下半层分层

NuttX 硬件看门狗走标准的 upper-half / lower-half 驱动分层：

  • 上半层 drivers/timers/watchdog.c：通用逻辑，提供 /dev/watchdogN 字符设备、ioctl 接口、automonitor、panic notifier。与具体硬件无关，所有 arch 共用。
  • 下半层 各 arch 的 *_wdog.c（如 arch/arm/src/am335x/am335x_wdog.c、arch/arm/src/imxrt/imxrt_wdog.c、arch/arm/src/kinetis/kinetis_wdog.c）：操作具体硬件寄存器，实现 watchdog_ops_s 操作集。

3.2 下半层操作集 watchdog_ops_s（include/nuttx/timers/watchdog.h:150）

struct watchdog_ops_s
{
  /* 必需方法 */
  int  (*start)(struct watchdog_lowerhalf_s *lower);      /* 启看门狗, 重置定时到当前 timeout */
  int  (*stop)(struct watchdog_lowerhalf_s *lower);       /* 停看门狗 */

  /* 可选方法 */
  int  (*keepalive)(struct watchdog_lowerhalf_s *lower);  /* "喂狗"/"ping", 重置定时但不停 */
  int  (*getstatus)(struct watchdog_lowerhalf_s *lower, struct watchdog_status_s *status);
  int  (*settimeout)(struct watchdog_lowerhalf_s *lower, uint32_t timeout);  /* 设新超时(并重置) */
  xcpt_t (*capture)(struct watchdog_lowerhalf_s *lower, xcpt_t handler);     /* 改超时行为: 不复位而调用户回调 */
  int  (*ioctl)(struct watchdog_lowerhalf_s *lower, int cmd, unsigned long arg);
};

下半层 start() 把硬件看门狗外设的复位倒计时打开——你必须在倒计时归零前调 keepalive() 喂狗重置它。如果倒计时归零，硬件外设直接拉复位线，整个芯片复位，内核什么也来不及做。这是硬件看门狗的本职，也是它区别于软狗的根本。

3.3 上半层状态结构 watchdog_upperhalf_s（watchdog.c:94）

struct watchdog_upperhalf_s
{
#ifdef CONFIG_WATCHDOG_PANIC_NOTIFIER
  struct notifier_block nb;       /* panic 时关狗的 notifier 注册块 */
#endif
#ifdef CONFIG_WATCHDOG_AUTOMONITOR
  /* automonitor 喂狗源实例(按编译开关选其一) */
  bool monitor;                   /* automonitor 是否正在运行 */
#endif
  uint8_t   crefs;                /* 设备被 open 的次数 */
  mutex_t   lock;                 /* 互斥 */
  FAR char *path;                /* 注册路径 "/dev/watchdog0" */
  FAR struct watchdog_lowerhalf_s *lower;  /* 下半层实例 */
};

上半层是下半层的"包装器"：负责文件接口、ioctl 分发、automonitor 调度、panic 注册；下半层只管硬件寄存器。

3.4 用户接口：ioctl 命令表

通过 /dev/watchdog0 字符设备，ioctl 命令（watchdog.c 各 case）：

ioctl 行号 作用
WDIOC_STARTwatchdog.c:550启动看门狗（先停 automonitor，再调 lower->ops->start）
WDIOC_STOP（write 'V' 同义，watchdog.c:483）停看门狗
WDIOC_KEEPALIVEwatchdog.c:669喂狗——重置定时但不停，防 imminent 复位
WDIOC_SETTIMEOUTwatchdog.c:614设新超时（毫秒），并重置
WDIOC_GETSTATUS（status ioctl）取 watchdog_status_s：flags/timeout/timeleft
WDIOC_CAPTUREwatchdog.c:635关键：注册用户回调，超时时不复位而调它

watchdog.c:483 还实现 Linux 风格的"写 'V' 关狗"——往设备写一个字符 'V' 表示关看门狗，魔法字符驱动标准做法，与 Linux 内核 /dev/watchdog 兼容。

3.5 watchdog_status_s（include/nuttx/timers/watchdog.h:137）

struct watchdog_status_s
{
  uint32_t flags;     /* WDFLAGS_* 状态位 */
  uint32_t timeout;   /* 当前超时设置(毫秒) */
  uint32_t timeleft;  /* 距超时剩余时间(毫秒) */
};

用户调 WDIOC_GETSTATUS 取此结构，可知看门狗当前超时设置和剩余时间。flags 中 WDFLAGS_ACTIVE 表示已启动、WDFLAGS_CAPTURE 表示已 capture 改回调模式。

4. 硬件看门狗"超时"会发生什么 —— 三种行为

这是硬件看门狗的核心问题。超时后果由下半层硬件能力 + 配置决定，有三种可能：

4.1 行为 A：硬件复位 MCU（默认）

这是看门狗的本职。下半层 start() 把硬件看门狗外设的复位倒计时打开，你必须在倒计时归零前调 keepalive() 喂狗重置它。如果倒计时归零，硬件外设直接拉复位线，整个芯片复位，内核什么也来不及做。

watchdog_ops_s 注释明说："Start the watchdog timer, resetting the time to the current timeout"——下半层 start 就是把硬件看门狗跑起来。超时后硬件自己复位，根本不经过内核。这是真看门狗该有的行为：不依赖任何软件执行，纯硬件复位。即使内核完全死锁、中断不再响应、调度器卡死，硬件看门狗外设的倒计时仍在自减，归零就拉线——这是它能"保命"的根本原因。

4.2 行为 B：调用用户回调而不复位（WDIOC_CAPTURE）

watchdog.c:629 的 WDIOC_CAPTURE 逻辑：

case WDIOC_CAPTURE:
  {
    FAR struct watchdog_capture_s *capture = (FAR void *)arg;
    /* Don't reset on watchdog timer timeout; instead, call this user
     * provider timeout handler. NOTE: Providing handler==NULL will
     * restore the reset behavior. */
    if (lower->ops->capture)   /* 可选方法 */
      {
        xcpt_t old = lower->ops->capture(lower, capture->newhandler);
        capture->oldhandler = old;
      }
  }

调 WDIOC_CAPTURE 传一个 xcpt_t handler，下半层就把硬件看门狗的复位行为改成"超时产生中断并跳到你的 handler"。前提是底层硬件支持——有些看门狗外设只能复位不能改行为，那 ops->capture 就是 NULL，ioctl 返回 -ENOSYS。

handler == NULL 时恢复复位行为。这是行为 A ↔ 行为 B 的切换开关。

watchdog_capture_s（include/nuttx/timers/watchdog.h:127）：

struct watchdog_capture_s
{
  xcpt_t newhandler;  /* 新 capture 处理器 */
  xcpt_t oldhandler;  /* 旧 capture 处理器(返回用) */
};

典型用途：看门狗超时中断里做紧急日志、关盘、dump，然后再复位——给系统一个"临终遗言"窗口。但这个窗口极短，取决于具体硬件看门狗外设是否在 capture 模式下还能再喂一次推迟复位。

4.3 行为 C：触发 notifier chain（CONFIG_WATCHDOG_TIMEOUT_NOTIFIER）

watchdog.c:152 定义 notifier chain：

#ifdef CONFIG_WATCHDOG_TIMEOUT_NOTIFIER
static ATOMIC_NOTIFIER_HEAD(g_watchdog_notifier_list);
#endif

watchdog.c:763 的 watchdog_automonitor_timeout：

void watchdog_automonitor_timeout(void)
{
  atomic_notifier_call_chain(&g_watchdog_notifier_list, action, data);
}

这是给下半层的"超时中断处理函数"里调的——硬件看门狗超时（在还没复位前的那段窗口，通常配合 capture 模式使用）会调用 notifier chain 上注册的所有回调。这样内核子系统能在复位前的最后窗口做点记录、dump、关盘等动作。

注册/注销接口（watchdog.c:720）：

  • watchdog_notifier_chain_register(nb) — 注册一个 notifier
  • watchdog_notifier_chain_unregister(nb) — 注销

5. automonitor：内核替你喂狗

5.1 为什么需要 automonitor

应用代码手动 WDIOC_KEEPALIVE 喂狗的问题：应用卡死就忘了喂，狗就叫——这正是看门狗该有的行为。所以你不应该在应用里喂狗。

但有些场景你希望看门狗"只在内核某些特定子系统卡死时才复位"——比如只要调度器还在跑、只要工作线程还在跑、只要 idle 还在进——而不是只要应用卡就复位。CONFIG_WATCHDOG_AUTOMONITOR 就是这个机制：让内核用另一个独立的定时源去喂狗，如果那个定时源也卡了，狗就复位。

设计哲学：用独立定时源喂狗 = 监控那个定时源的活性。喂狗源卡死→狗没人喂→狗复位。换喂狗源就是换"被监控对象"。

5.2 六种 automonitor 喂狗源（watchdog.c:74-86）

通过 CONFIG_WATCHDOG_AUTOMONITOR_BY_* 编译开关选其一：

配置 喂狗源 喂狗源卡了狗才叫 监控对象
AUTOMONITOR_BY_ONESHOToneshot 单次定时器oneshot 中断不触发arch oneshot timer 硬件 + 中断响应
AUTOMONITOR_BY_TIMERPOSIX timer 周期定时器timer 中断不触发timer 子系统 + 中断响应
AUTOMONITOR_BY_WDOG软件看门狗 wdogwdog 回调不跑调度器/tick 中断（最常用）
AUTOMONITOR_BY_WORKERwork_queue 工作线程工作线程不跑work_queue 子系统（LPWORK 线程活性）
AUTOMONITOR_BY_CAPTUREcapture 中断喂狗capture 中断不触发硬件看门狗中断响应链

watchdog.c:60-66 定义喂狗间隔：

#  if !defined(CONFIG_WATCHDOG_AUTOMONITOR_BY_CAPTURE) && \
      !defined(CONFIG_WATCHDOG_AUTOMONITOR_BY_IDLE)

默认喂狗间隔 = 硬件狗超时的一半——留一半余量，避免中断延迟导致喂狗不及时但狗仍复位。capture/idle 模式不需要此间隔（它们是事件驱动而非周期驱动）。

5.3 经典例子：AUTOMONITOR_BY_WDOG 软狗喂硬狗

watchdog.c:208：

static void watchdog_automonitor_wdog(wdparm_t arg)
{
  FAR struct watchdog_upperhalf_s *upper = (FAR void *)arg;
  FAR struct watchdog_lowerhalf_s *lower = upper->lower;
  if (upper->monitor)
    {
      lower->ops->keepalive(lower);    /* 喂硬件狗 */
      wd_start(&upper->wdog, WATCHDOG_AUTOMONITOR_PING_INTERVAL_TICK,
               watchdog_automonitor_wdog, (wdparm_t)upper);  /* 再续软狗 */
    }
}

6. panic notifier：内核 panic 时关狗

6.1 机制（watchdog.c:96, 330）

#ifdef CONFIG_WATCHDOG_PANIC_NOTIFIER
  /* When a crash occurs, stop the watchdog */
  struct notifier_block nb;
#endif

watchdog.c:330 的 wdog_notifier 注册到内核 panic notifier chain：

static int wdog_notifier(FAR struct notifier_block *nb, unsigned long action,
                         FAR void *data)
{
  FAR struct watchdog_upperhalf_s *upper = (FAR struct watchdog_upperhalf_s *)nb;
  if (action == PANIC_KERNEL)
    {
#ifdef CONFIG_WATCHDOG_AUTOMONITOR
      watchdog_automonitor_stop(upper);   /* 停 automonitor 喂狗 */
#else
      return upper->lower->ops->stop(upper->lower);  /* 直接停硬狗 */
#endif
    }
}

注册发生在 watchdog_register()（watchdog.c:860）：

upper->nb.notifier_call = wdog_notifier;
panic_notifier_chain_register(&upper->nb);

6.2 意图与代价

意图：内核已经 panic 在打 dump 了，你不想这时候硬件狗复位把 dump 输出给吃了。启用 CONFIG_WATCHDOG_PANIC_NOTIFIER 就会在 panic 时把狗停掉，让 dump 输出完整。

代价：panic 后系统不复位，得人工复位或外部硬复位——这是开发期配置，生产期一般不开。生产期你希望 panic 后复位重启服务，而非挂在 dump 上。

配置 panic 后行为 适用场景
无 CONFIG_WATCHDOG_PANIC_NOTIFIERpanic → dump 输出被打断 → 硬狗复位 → 重 boot生产期
CONFIG_WATCHDOG_PANIC_NOTIFIERpanic → 停狗 → dump 完整输出 → 等人工复位开发期调试

7. 软狗与硬狗的联动与衔接

7.1 AUTOMONITOR_BY_WDOG 是唯一联动点

软狗与硬狗唯一的直接联动点是 CONFIG_WATCHDOG_AUTOMONITOR_BY_WDOG 模式（本文 §5.3）：软狗回调跑硬狗 keepalive。其他 automonitor 源都不用软狗。

7.2 timer_wdog.c 的另一层联动：POSIX timer 用软狗

drivers/timers/timer_wdog.c 是 POSIX timer 的软狗后端实现。timer_wdog_dev_s（timer_wdog.c:42）：

struct timer_wdog_dev_s
{
  struct timer_lowerhalf_s lower;   /* 上半层 timer 接口 */
  struct wdog_s            wdog;    /* 软狗实例 */
  atomic_t                 period;  /* timer 周期 */
  spinlock_t               lock;
  tccb_t                   callback; /* timer 上半层回调 */
  FAR void                *arg;
};

timer_wdog_start（timer_wdog.c:94）用 wd_start 启软狗；到期回调 timer_wdog_handler（timer_wdog.c:244）调上半层 timer 的 callback——即 POSIX timer 到期发信号给 owner。

这是软狗被当作 timer 子系统的底层驱动，与硬件看门狗无关。但结构上和硬件看门狗 automonitor BY_WDOG 模式同源：都是"用软狗驱动另一个东西"。

  • 软件看门狗（wdog）不是看门狗——它是 NuttX 内核用的"按 tick 排序的软件定时器链表"，到期在中断上下文调你给的回调函数，不复位任何东西。POSIX timer、信号超时、mqueue 超时、mutex 超时都靠它。三种底层模式：普通 tick、tickless、hrtimer。
  • 硬件看门狗（watchdog）才是真看门狗——上半层 drivers/timers/watchdog.c + 下半层各 arch *_wdog.c，驱动片上硬件外设，默认超时硬件复位 MCU。可用 WDIOC_CAPTURE 改成调用户回调，或 CONFIG_WATCHDOG_TIMEOUT_NOTIFIER 触发 notifier chain。
  • automonitor 六种喂狗源让内核用独立定时源替你喂狗：源卡了狗才复位。BY_WDOG 模式下软狗喂硬狗是经典联动——软狗调度卡死→硬狗没人喂→硬狗复位 MCU。
  • panic notifier 在 panic 时停狗保 dump 完整，是开发期配置；生产期不开让 panic 后复位重启。
  • 软狗与硬狗本质不同但名字混用，阅读源码必须先看路径才能判断在讲哪一个。
