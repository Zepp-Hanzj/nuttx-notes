NuttX 消息队列与任务间通信详解

本文针对 NuttX 深入讲解：

  • 消息队列（Message Queue）的两类实现：POSIX mqueue（mq_*）与 System V msg（msgget/msgsnd/msgrcv）；
  • 任务间通信（IPC）的核心机制：数据结构、send/receive 全流程、等待队列、超时回收；
  • 与信号量/互斥锁的对比，以及选型建议。

全部源码引用对应本仓库（nuttx/）里的真实文件，行号即真实行号。

1. 概览：NuttX 里的"消息队列"指什么

NuttX 里"消息队列"不是单一接口，而是两套并存的 IPC 机制：

接口族 标准来源 API 内核实现 用途
mqueue POSIX.1-2001 mq_open/mq_send/mq_receive/mq_close/mq_unlink/mq_notify nuttx/fs/mqueue/ + nuttx/sched/mqueue/ 命名消息队列，文件系统风格（/dev/mqueue），带优先级
msg System V msgget/msgsnd/msgrcv/msgctl nuttx/sched/mqueue/msg*.c 按 key_t 索引的消息队列，带消息类型 mtype

两者都建在共同的 mqueue_cmn_s 数据结构上，差异仅在"命名/索引方式"和"是否带 mtype"。

1.1 与信号量/互斥锁的定位差异

维度 信号量 Semaphore 互斥锁 Mutex 消息队列 Message Queue
语义 计数同步，传达"事件发生了" 二值互斥，保护临界区 数据传输，把一块字节从 A 送 B
携带数据 无，仅 count 无 有，可带任意字节 + 优先级/mtype
同步方式 异步通知（post 唤醒） 持者独占 阻塞/非阻塞/超时三种
多对多 多任务等一 sem 单持单等 多 send 多 receive，队头队尾
谁释放 任何任务可 post 只持者 unlock 任任务可 send/receive
一句话：信号量传"事件"、互斥锁守"资源"、消息队列搬"数据"。

2. 数据结构

2.1 共用基础 mqueue_cmn_s

nuttx/include/nuttx/mqueue.h line 103：

struct mqueue_cmn_s
{
  dq_queue_t waitfornotempty;  /* 等"队非空"的任务队列(MQ_WNELIST) */
  dq_queue_t waitfornotfull;   /* 等"队非满"的任务队列(MQ_WNFLIST) */
  int16_t   nwaitnotfull;      /* 等非满的任务数(快路径判定用) */
  int16_t   nwaitnotempty;     /* 等非空的任务数 */
};

两条等待队列是消息队列的"调度接口"：

  • MQ_WNELIST —— receive 阻塞时挂这里，等"队非空"（有消息可取）
  • MQ_WNFLIST —— send 阻塞时挂这里，等"队非满"（有空位可塞）

二者都是 dq_queue_t（双链优先队列），按 sched_priority 降序，队头永远是优先级最高的等待者——post/send 唤醒时取队头，即"优先级高的先拿到/先让位"。

2.2 POSIX mqueue 主体 mqueue_inode_s

nuttx/include/nuttx/mqueue.h line 113：

struct mqueue_inode_s
{
  struct mqueue_cmn_s  cmn;        /* 共用基础 */
  FAR struct inode    *inode;      /* 所属 inode(命名 mqueue 走 fs 树) */
  struct list_node     msglist;    /* 消息链表表头(按优先级插) */
  int16_t              maxmsgs;    /* 队列最大消息数 */
  int16_t              nmsgs;      /* 当前消息数 */
#if CONFIG_MQ_MAXMSGSIZE < 256
  uint16_t             maxmsgsize; /* 单条消息最大字节数 */
#endif
  ...
};

msglist 是 list_node 单链表，消息按优先级降序插入——mq_send 时按 msg 的 priority 找位置插，mq_receive 时取队头（优先级最高的那条）。

2.3 消息节点 mqueue_msg_s

每条消息是一个挂在 msglist 上的节点：

struct mqueue_msg_s
{
  struct list_node  node;        /* 链表节点 */
  uint8_t            priority;   /* mq_send 时指定,0..MQ_PRIO_MAX */
  uint16_t           size;       /* 实际消息字节数 */
  char               msg[CONFIG_MQ_MAXMSGSIZE]; /* 数据缓冲 */
};

3.3 关键观察

  • send/receive 是对称的：send 阻塞在"队满"，receive 阻塞在"队空"；各自唤醒对方的等待队列。
  • 优先级双轨：消息按 priority 降序在 msglist 里（高优消息先被收），等待任务按 sched_priority 降序在等待队列里（高优任务先被唤醒）。
  • ISR 可发消息：send 全程关 IRQ 临界区，因 ISR 里调 mq_send 是合法用法（常用于 ISR→任务的事件投递）。

4. 超时、信号回收：nxmq_wait_irq

mq_waitirq.c —— 等消息的任务被超时或信号打断时，必须把占的等待队列项摘掉、补 errcode、唤醒，否则"漏唤醒"或"漏计数"。

4.1 完整逻辑

void nxmq_wait_irq(struct tcb_s *wtcb, int errcode)
{
  msgq = wtcb->waitobj;             // 拿到所等的 mqueue
  DEBUGASSERT(msgq);

  // 按当前状态摘对应等待队列
  if (wtcb->task_state == TSTATE_WAIT_MQNOTEMPTY)
    {
      msgq->cmn.nwaitnotempty--;                      // 补计数
      dq_rem(wtcb, MQ_WNELIST(msgq->cmn));            // 摘离 WNE 队列
    }
  else  // TSTATE_WAIT_MQNOTFULL
    {
      msgq->cmn.nwaitnotfull--;
      dq_rem(wtcb, MQ_WNFLIST(msgq->cmn));
    }

  wtcb->waitobj = NULL;
  wtcb->errcode = errcode;           // ETIMEDOUT / EINTR / ECANCELED

  if (nxsched_add_readytorun(wtcb))  // 唤醒回就绪队列
    up_switch_context(this_task(), rtcb);  // 若队头变则 svc 切
}

4.2 与 nxsem_wait_irq 的同构

把 §4.1 和信号量文档 §8.1 对照，二者结构完全一致——摘队列、补 errcode、唤醒。这是 NuttX "等资源任务被打断"的统一回收路径，所有 IPC 子系统（sem/mqueue/msg）都走同一套范式。

差异仅在于"队列有几条"：

  • sem/mutex：1 条 waitlist
  • mqueue：2 条 WNE/WNF（按 task_state 分流）

5. 任务退出回收：nxmq_recover

mq_recover.c —— 任务意外退出时若还在等消息，或挂着未取的消息，需回收防"漏队列项"导致计数永错。

5.1 完整逻辑

void nxmq_recover(struct tcb_s *tcb)
{
  // 若该任务在等消息,走 wait_irq 回收(补 ECANCELED)
  if (tcb->task_state == TSTATE_WAIT_MQNOTEMPTY ||
      tcb->task_state == TSTATE_WAIT_MQNOTFULL)
    {
      nxmq_wait_irq(tcb, ECANCELED);
    }

  // POSIX mqueue 还要释放该任务打开的所有 mqdes 文件描述符
  // (走 fs 关闭路径,见 mq_close.c 的 closeatus == TCB 文件表清空)
}

5.2 防的什么漏洞

若不回收 后果
等消息的任务死了，WNE/WNF 队列里残留项 nwaitnotempty/nwaitnotfull 计数永错，send/receive 的快路径判定失真
持有 mqdes 的任务死了，fd 不释放 mqdes 表泄漏，mq_unlink 时 inode 引用计数不归 0
通知注册的任务死了，ntpid 留 mq_notify 永返回 EBUSY，新任务无法注册

跟 nxsem_recover 同构——任务退出时 IPC 子系统都靠这套兜底防孤儿挂。

6. System V msgqueue 的差异

msgget/msgsnd/msgrcv/msgctl 在共用 mqueue_cmn_s 之上加了三个特性：

6.1 key_t 索引

msgget(key, flags) —— 用 key_t（整数）查全局 g_msgq_table 表找 msgq；找不到且 IPC_CREAT 就新建。这跟 POSIX mqueue 的"文件系统路径名"索引方式不同，System V 风范。

索引方式 POSIX mqueue System V msgqueue
键 路径名 /foo 整数 key_t = 0x12345
全局可见 走 fs 树，任何进程可 mq_open(name) 走 g_msgq_table，任何进程可 msgget(key)
销毁 mq_unlink(name) 且所有 fd 关闭 msgctl(id, IPC_RMID)

6.2 mtype 类型过滤

msgrcv(msgid, buf, size, mtype, flags) 的 mtype 参数支持四种过滤：

mtype 行为
== 0 取队头（任意类型，实际按 priority 序）
> 0 取指定类型的第一条
< 0 取类型 ≤ |mtype| 的第一条（最低类型优先）
MSG_EXCEPT flag + mtype 取类型 ≠ mtype 的第一条

实现里 msgrcv 遍历 msglist 找匹配 mtype 的节点，而非简单取队头。这让"多类型消息混用一队"成为可能——比 POSIX mqueue 的纯优先级序更灵活。

典型用法：控制消息 mtype=1，数据消息 mtype=2，接收方按需取。

6.3 msgctl 控制

msgctl(msgid, cmd, buf) 支持三个命令：

cmd 作用 对应 POSIX mqueue
IPC_STAT 取队列属性（nmsgs/maxmsgs/uid/gid…）到 buf mq_getattr（仅 O_NONBLOCK）
IPC_SET 改队列属性（uid/gid/mode） 无
IPC_RMID 删除队列 mq_unlink

POSIX mqueue 没这套改属性的，因走 fs 树，改属性用 chmod/chown。

7. mq_notify：异步通知

POSIX mqueue 独有的 mq_notify——注册"队列从空变非空时给我发个信号"，实现真异步通知。让高优任务不阻塞等消息，干别的，空变非空时信号打断它去取。

7.1 注册：mq_notify

mq_notify(mqd_t mqdes, const struct sigevent *notification);

mq_notify.c line 97 实现要点：

rtcb = this_task();
msgq = inode->i_private;

  // 无现注册,notification != NULL 则注册
  memcpy(&msgq->ntevent, notification, sizeof(struct sigevent));
  msgq->ntpid = rtcb->pid;         // 记下注册者 PID
}
else if (msgq->ntpid != rtcb->pid || notification != NULL)
{
  // 已被别人注册,或自己重复注册 → EBUSY
  errval = EBUSY;
}
else
{
  // 自己持 notification==NULL → 解注册
  memset(&msgq->ntevent, 0, sizeof(struct sigevent));
  msgq->ntpid = INVALID_PROCESS_ID;
  nxsig_cancel_notification(&msgq->ntwork);
}
leave_critical_section(flags);

关键约束：每个 mqueue 只能挂一个通知注册（再注册返回 EBUSY）；只有注册者自己能解注册。

7.2 触发：nxmq_notify_send

mq_sndinternal.c line 239 —— mq_send 把消息塞进队后，若 nmsgs 从 0→1（空变非空），触发通知：

void nxmq_notify_send(struct mqueue_inode_s *msgq)
{
  if (msgq->ntpid != INVALID_PROCESS_ID)
  {
    // 抄一份 sigevent,清注册(一次性)
    memcpy(&event, &msgq->ntevent, sizeof(struct sigevent));
    pid = msgq->ntpid;
    memset(&msgq->ntevent, 0, sizeof(struct sigevent));
    msgq->ntpid = INVALID_PROCESS_ID;

    // 给目标 pid 投信号
    nxsig_notification(pid, &event, SI_MESGQ, &msgq->ntwork);
  }

  // 接着唤醒等"非空"的阻塞等者(WNE 队头)
  if (msgq->cmn.nwaitnotempty > 0)
  {
    btcb = dq_remfirst(MQ_WNELIST(msgq->cmn));
    wd_cancel(&btcb->waitdog);
    btcb->waitobj = NULL;
    nxsched_add_readytorun(btcb);  // 唤醒
    up_switch_context(...);
  }
}

7.3 通知是一次性的

注释明写——通知触发后自动解注册（ntpid = INVALID_PROCESS_ID）。要继续通知必须重新调 mq_notify。这跟 POSIX 规范一致：防止"队列持续非空时反复投信号"。

7.4 通知与阻塞等的优先关系

nxmq_notify_send 里先发通知、再唤醒阻塞等者。但 POSIX 规范要求"若有阻塞等者，通知不触发"——NuttX 注释明说它故意不遵守这条（line 86-93）：即"通知总会发，等者也被唤醒"。理由是 NuttX 信号投递是异步的，通知和唤醒并不互斥，实现简单。

8.3 优先级双轨

消息队列的"优先级"分两个独立轨道：

  • 消息优先级 priority —— msglist 队头是最高 priority 的消息，mq_receive 取它
  • 等待者优先级 sched_priority —— WNE/WNF 队头是最高 sched_priority 的等者，send/receive 唤它

两条轨道独立：高优任务等低 priority 消息，照样先被唤醒；低优任务等高 priority 消息，也只能干等。这跟"按需分配"不同，是严格优先级序。

8.4 与任务调度文档的接合

临界区内的 nxsched_add_readytorun 若返回 true（队头变），会触发 up_switch_context —— 这对应任务调度文档 §5 SVCall 路径：svc 同步切到新队头任务。若 sched_lock 期间，则进 g_pendingtasks 状态置 TSTATE_TASK_PENDING（任务调度文档 §3.3），等 sched_unlock 时 nxsched_merge_pending 并入就绪队列才真切。

ISR 里 mq_send 唤醒高优等者时，临界区内的 nxsched_add_readytorun 不真切，只 pend PendSV —— 等所有高优 IRQ 退完后进 PendSV handler 才真切（任务调度文档 §5 PendSV 最低优先级机制）。

小结一句话：NuttX 消息队列有两套接口——POSIX mqueue（路径名索引、消息按 priority 排序、支持 mq_notify 异步通知）和 System V msgqueue（key_t 索引、消息带 mtype 支持按类型过滤接收），二者共用 mqueue_cmn_s 基础结构和 WNE/WNF 双等待队列。send/receive 完全对称——队满时 send 阻塞在 WNF、队空时 receive 阻塞在 WNE，各自唤醒对方。全程关 IRQ 临界区保护（ISR 可合法 mq_send），真切换延后到 svc/PendSV。异常路径回收（nxmq_wait_irq 超时/信号、nxmq_recover 任务退出）与信号量完全同构，是 NuttX IPC 子系统的统一范式。选型一句话：信号量传事件、互斥锁守资源、消息队列搬数据。
